# Полный курс GitLab CI/CD — Подробное руководство на русском языке

---

## Введение в курс

Это курс по **GitLab CI/CD** — встроенной платформе непрерывной интеграции и доставки, которая позволяет автоматизировать тестирование, сборку контейнеров и развёртывание приложений прямо из репозитория. Курс охватывает всё: от базовых принципов CI/CD до продвинутых возможностей — многоразовые шаблоны, self-managed раннеры, управление секретами и Auto DevOps.

Структура каждого модуля:
1. **Теоретическая лекция** — концепции и терминология
2. **Живая демонстрация** — пошаговая реализация пайплайнов в реальном времени
3. **Практическая лаборатория** — интерактивные задачи прямо в браузере (на платформе KodeKloud, без локальной установки)

## Часть 1. Базовые концепции CI/CD

### 1.1 Что такое CI/CD и зачем это нужно

**CI/CD** расшифровывается как **Continuous Integration / Continuous Delivery (или Deployment)** — непрерывная интеграция и непрерывная доставка/развёртывание. Это набор практик, которые автоматизируют цикл разработки: от написания кода до его попадания в production.

Без CI/CD команды сталкиваются с типичными проблемами:

| Проблема | Последствие |
|---|---|
| Отсутствие автоматического тестирования | Баги попадают в production |
| Ручные ошибки при деплое | Несоответствие окружений, неверные конфиги |
| Медленная обратная связь | Задержки в исправлении ошибок, долгие циклы релизов |

### 1.2 Типичный Git-воркфлоу

Вся работа строится вокруг Git-репозитория, который хранится на таких платформах, как GitLab, GitHub или Bitbucket. Стандартный процесс выглядит так:

1. Разработчик создаёт **feature-ветку** от защищённой ветки `main`
2. Вносит изменения и пушит их в свою ветку
3. Открывает **Merge Request (MR)** — запрос на слияние с `main`, что запускает ревью кода
4. После одобрения изменения вливаются в `main` и деплоятся в целевое окружение

> **Важная практика:** включайте защиту ветки `main`, чтобы слияние было возможно только через MR и только после прохождения автоматических проверок.

### 1.3 Непрерывная интеграция (Continuous Integration)

CI гарантирует, что каждое изменение кода автоматически тестируется и проверяется **до** слияния в `main`. Процесс выглядит так:

1. Разработчик создаёт ветку `feature-A` от `main`
2. Открывает MR — это автоматически запускает CI-пайплайн, который выполняет:
   - Юнит-тесты
   - Проверку зависимостей и лицензий
   - Сборку артефактов
   - Статический анализ кода и сканирование на уязвимости
3. Если хоть один этап падает — MR остаётся открытым, разработчик фиксит проблему и пушит снова (пайплайн перезапускается)
4. После прохождения всех проверок MR одобряется и вливается в `main`
5. После слияния запускается ещё один пайплайн — интеграционные тесты для обновлённой ветки `main`

Параллельно другой разработчик может работать над `feature-B`. После слияния обеих веток пост-мёрж пайплайн проверяет, что A и B не конфликтуют между собой — `main` остаётся стабильной.

> **Ключевое преимущество CI:** быстрая обратная связь помогает обнаруживать проблемы на ранних стадиях и снижает количество конфликтов при слиянии.

### 1.4 От интеграции к доставке и развёртыванию

После того как CI подтверждает качество кода, CD автоматизирует путь в production:

| Процесс | Целевое окружение | Нужно ли ручное подтверждение | Типичный сценарий |
|---|---|---|---|
| Continuous Delivery (непрерывная доставка) | Non-production (staging и др.) | Да | Ручное одобрение перед production |
| Continuous Deployment (непрерывное развёртывание) | Production | Нет | Полностью автоматические релизы |

**Continuous Delivery:**
- Деплой в staging или QA-окружения
- Запуск интеграционных, нагрузочных и end-to-end тестов
- Ожидание ручного одобрения для production

**Continuous Deployment:**
- Автоматический деплой в production после успешного CI
- Идеально для команд с зрелым набором тестов и высокой уверенностью в качестве

> ⚠️ **Предупреждение:** отказ от ручного одобрения в CD требует исчерпывающего набора автоматических тестов. Без этого вы рискуете production-инцидентами.

### Выводы по разделу 1

- CI автоматизирует тестирование и валидацию для каждого изменения кода
- CD добавляет автоматический деплой в staging с человеческим "шлюзом"
- Continuous Deployment полностью автоматизирует релизы в production
- Вместе CI/CD ускоряет циклы релизов, повышает качество ПО и минимизирует операционные риски

## Часть 2. Реальная задача: команда Dasher Technology

### 2.1 Контекст и боли заказчика

Компания **Dasher Technology** предоставляет платформу, объединяющую данные, приложения и устройства в on-premises окружениях. Их R&D команда исследует стратегию миграции в облако и контейнеризацию — начиная с Node.js проекта, с последующим расширением на Java и Python приложения.

**Alice** возглавляет новую DevOps-команду, которой поручено построить CI/CD пайплайн с нуля. Архитектура будет multi-cloud, с использованием **Docker** для контейнеризации и **Kubernetes** для оркестрации.

Первоначальная оценка Алисой ситуации выявила следующие болевые точки:

- Нет системы контроля версий и управления ветками
- Ручная интеграция кода и тестирование
- Медленные и ненадёжные тесты с низким покрытием
- Редкие слияния, что повышает риски при релизе
- Ручные деплои в dev, staging и production окружения

**Цель:** реализовать CI/CD пайплайн, состоящий из:
1. Контроля версий и совместной работы на GitLab
2. Автоматических юнит-тестов с покрытием кода
3. Сборки Docker-образов и их публикации в registry
4. Развёртывания в Kubernetes-кластеры
5. Автоматических интеграционных тестов перед выкаткой в production

### 2.2 Сравнение CI/CD инструментов

Команда оценила несколько популярных платформ:

| Инструмент | Плюсы | Минусы |
|---|---|---|
| Jenkins | Высокая расширяемость, большая экосистема плагинов | Сложная настройка, нужно управлять инфраструктурой |
| GitLab CI/CD | Интегрирован с GitLab, автоматическое масштабирование раннеров | Меньше community-плагинов, чем у Jenkins |
| Travis CI | Простая YAML-конфигурация | Ограниченная параллельность, медленный enterprise-уровень |
| CircleCI | Быстрая работа с контейнерами, поддержка Docker | Ограничения на бесплатном тарифе |
| GitHub Actions | Нативен для GitHub, богатый marketplace | Требует экосистемы GitHub |
| Spinnaker | Продвинутые стратегии деплоя | Крутая кривая обучения |
| Bamboo | Тесная интеграция с Atlassian | Лицензионные расходы, меньше community |

### 2.3 Почему Jenkins не подошёл

Алиса изначально выбрала **Jenkins** — из-за его зрелости и активного сообщества. Однако настройка и обслуживание Jenkins-сервера даже для одного Node.js проекта оказалась крайне трудозатратной:

- Развернуть виртуальные машины с достаточными CPU, памятью и диском
- Установить Java JDK, Jenkins и нужные плагины
- Настроить правила файрвола и группы безопасности
- Установить и поддерживать несколько версий Node.js и npm
- Установить Docker для сборки контейнеров
- Добавить инструменты Kubernetes: `kubectl`, Helm
- Интегрировать внешние CLI для тестирования и отчётности

А расширение этой инфраструктуры для Java или Python на AWS/Azure добавляет ещё больше сложности:
- Java: Maven или Gradle
- Python: virtualenv, pip
- Cloud CLI: AWS CLI, Azure CLI
- DevSecOps: сканеры уязвимостей — Trivy и KubeSec

> **Ключевой вывод:** управление собственной CI/CD инфраструктурой означает трату времени на настройку и обслуживание вместо написания пайплайнов.

Поскольку многие члены команды были новичками в этой экосистеме инструментов, Алисе было нужно решение, которое:
- Требует минимальной настройки инфраструктуры
- Позволяет команде сосредоточиться на написании пайплайнов
- Обеспечивает встроенную масштабируемость и безопасность

**Итог:** Алиса выбрала **GitLab CI/CD**.

## Часть 3. Что такое GitLab CI/CD

### 3.1 Общая концепция

**GitLab Server** — это open-source DevOps платформа, которая объединяет весь жизненный цикл разработки ПО — контроль версий, отслеживание задач, CI/CD пайплайны, реестры пакетов и многое другое — в единый интерфейс.

**GitLab CI/CD** — это встроенная система непрерывной интеграции и доставки, которая автоматически запускает задания пайплайна при событиях в репозитории (коммит, merge request и т.д.). Вы описываете воркфлоу в одном YAML-файле, а GitLab сам берёт на себя всё — от запуска окружений до отчётности о результатах.

### 3.2 Как выполняются задания CI/CD

Каждое задание (job) в пайплайне выполняется на **Runner** — лёгком агенте, который запускает задачи в изолированном окружении. GitLab предлагает два типа раннеров:

- **SaaS Runners** — хостятся GitLab.com
- **Self-Managed Runners** — запускаются на вашей собственной инфраструктуре

#### SaaS Runners

SaaS Runners включены по умолчанию для всех проектов на GitLab.com — никакой дополнительной настройки не требуется. В зависимости от требований к сборке можно выбирать:

| Тип раннера | Платформа | Сценарий использования |
|---|---|---|
| Linux Runners | Ubuntu, Alpine | Широкая поддержка языков и инструментов |
| Windows Runners (Beta) | Windows Server | Windows-специфичные сборки (PowerShell, .NET и др.) |
| macOS Runners (Beta) | macOS | Сборки для экосистемы Apple (Xcode, Swift, CocoaPods) |
| GPU-enabled Runners | Linux + GPU | Высокопроизводительные задачи (обучение ML-моделей, инференс) |

GitLab в SaaS-режиме управляет:
- Подготовкой свежей VM или контейнера для каждого задания
- Кэшированием зависимостей для ускорения последующих сборок
- Подробной отчётностью о статусе заданий и логах

**Преимущества автоматизации:**
- Ускоренные релизы — быстрее доставляете фичи и фиксы
- Меньше ручных ошибок — стабильные, повторяемые деплои
- Повышенное качество — проблемы выявляются раньше в пайплайне

### 3.3 Определение пайплайна: файл `.gitlab-ci.yml`

**Пайплайн** — это последовательность из одного или нескольких заданий, запускаемых событиями репозитория. Чтобы его определить, создайте файл `.gitlab-ci.yml` в корне проекта:

```yaml
# .gitlab-ci.yml
workflow:
  name: My Awesome App Pipeline
  rules:
    - if: $CI_COMMIT_BRANCH == 'main'

unit_test_job:
  parallel:
    matrix:
      - OPERATING_SYSTEM:
          - saas-linux-small-amd64
          - shared-windows
          - saas-macos-medium-m1
  tags:
    - $OPERATING_SYSTEM
  image: node:17-alpine3.14
  script:
    - npm install
    - npm test
```

**Разбор конфигурации:**

- `workflow.rules` — пайплайн запускается только для ветки `main`
- `unit_test_job` — название задания
- `parallel.matrix` — задание выполняется параллельно на трёх типах ОС: Linux, Windows и macOS
- `tags` — указывает, на каком раннере запускать задание
- `image` — Docker-образ для окружения (`node:17-alpine3.14`)
- `script` — команды, которые выполняются последовательно: `npm install`, затем `npm test`

При запуске GitLab подготавливает три изолированных окружения, выполняет скрипты по очереди и сообщает об успехе или неудаче. Пайплайн считается успешным только когда проходят **все** параллельные задания.

**Пример вывода в логах задания:**

```
$ git clone --depth 20 "$CI_REPOSITORY_URL" .
$ git checkout -f 7e0b6435c3  # detached HEAD (main)
$ npm install
> npm install completed
$ npm test
> npm test passed
Cleaning up project directory and file-based variables
Job succeeded
```

### 3.4 Просмотр логов и артефактов

Проверить логи заданий или скачать сгенерированные артефакты можно через интерфейс GitLab или REST API:

1. Перейдите в **CI/CD > Pipelines**
2. Выберите пайплайн для просмотра стадий и заданий
3. Нажмите на задание, чтобы увидеть вывод в консоли и скачать артефакты (отчёты о тестах, бинарники и т.д.)

## Часть 4. Создание учётной записи GitLab и квоты

### 4.1 Вход в GitLab

Для начала работы нужно зайти на страницу входа GitLab. Доступны следующие способы аутентификации:
- Email и пароль
- Google
- GitHub
- Bitbucket
- Salesforce

### 4.2 Дашборд и пространства имён (Namespaces)

После входа GitLab отображает все ваши проекты, сгруппированные по пространству имён. По умолчанию создаются два типа:

| Тип namespace | Описание |
|---|---|
| Personal namespace | Назван по вашему username, содержит только ваши проекты |
| Group namespace | Папка для связанных проектов с общими правами, настройками и участниками |

### 4.3 Проекты в GitLab

**Проект** в GitLab — это эквивалент репозитория на других платформах. Внутри каждого проекта можно:
- Хранить код и ассеты
- Отслеживать задачи, доски и merge requests
- Проводить ревью кода
- Строить, тестировать и деплоить с помощью встроенных CI/CD пайплайнов

> GitLab объединяет управление репозиторием, отслеживание задач и CI/CD в одном месте — без лишних настроек.

### 4.4 Квоты использования CI/CD

Для мониторинга потребления CI/CD перейдите в **Edit Profile > Usage Quotas**.

**Триальные аккаунты GitLab включают 400 compute units в месяц.** Compute units измеряют время работы заданий на shared runners GitLab.

Формула расчёта:

```
Compute usage = длительность задания × cost factor
```

**Cost factor зависит от:**
- Видимости проекта (public, internal, private, open source)
- Типа и размера раннера (Linux, Windows, macOS, GPU-enabled и т.д.)

Разные типы VM имеют разные cost factors — подробности смотрите в документации GitLab.

> ⚠️ **Внимание:** превышение месячной квоты compute units может привести к дополнительным расходам. Регулярно мониторьте использование и рассмотрите добавление дополнительных раннеров или оптимизацию длительности заданий.

## Итоговые выводы по всему курсу

**1. CI/CD решает реальные проблемы разработки.** Без автоматизации команды страдают от ручных ошибок, медленных релизов и багов в production. CI/CD устраняет эти боли системно.

**2. GitLab CI/CD — оптимальный выбор для GitLab-экосистемы.** В отличие от Jenkins, который требует тяжёлой инфраструктуры, GitLab CI/CD встроен в платформу, масштабируется автоматически и не требует ручной настройки серверов.

**3. Пайплайн описывается одним YAML-файлом.** Весь воркфлоу — тесты, сборка образов, деплои — определяется в `.gitlab-ci.yml`. Это делает конфигурацию прозрачной, версионируемой и воспроизводимой.

**4. SaaS Runners убирают операционную нагрузку.** GitLab сам подготавливает изолированные окружения, кэширует зависимости и отчитывается о результатах — команда фокусируется на пайплайнах, а не на инфраструктуре.

**5. CI/CD — это не только автоматизация, но и культура.** Частые слияния, быстрая обратная связь от тестов, обязательные проверки перед мёржем — всё это формирует культуру качества в команде.

**6. Квоты важны для планирования.** На бесплатном/триальном аккаунте есть ограничения в 400 compute units в месяц. При планировании пайплайнов важно учитывать cost factor выбранных раннеров.

# GitLab CI/CD — Архитектура, компоненты и практика. Подробное руководство

---

## Часть 1. Основные компоненты GitLab CI/CD

### 1.1 Обзор четырёх ключевых строительных блоков

Вся система GitLab CI/CD строится на четырёх фундаментальных концепциях:

| Компонент | Описание |
|---|---|
| Pipeline (Пайплайн) | Автоматизированный процесс, оркестрирующий задания и стадии |
| Stage (Стадия) | Логическая группа заданий (например, `build`, `test`, `deploy`) |
| Job (Задание) | Отдельная единица работы, выполняемая на раннере |
| Script (Скрипт) | Shell-команды, выполняемые заданием |

Понимание этих четырёх компонентов — это основа для построения любого CI/CD пайплайна в GitLab. Давайте разберём каждый из них подробно.

---

### 1.2 Pipeline (Пайплайн)

**Пайплайн** — это главный объект всей системы CI/CD. Он описывается через файл `.gitlab-ci.yml`, который должен находиться в корне репозитория. Если файл расположен в другом месте или отсутствует — GitLab его не обнаружит и пайплайн не запустится.

**Ключевые характеристики пайплайна:**

- **Определение:** YAML-файл с именем `.gitlab-ci.yml`
- **Именование:** директива `workflow:name` позволяет задать кастомное имя, которое отображается во вкладке Pipelines
- **Триггеры:** пайплайны запускаются при `push`, `merge_request`, по расписанию (`schedule`) или вручную
- **Условия:** директива `rules` контролирует, когда именно запускать пайплайн

**Пример: запуск пайплайна только при коммитах в `main`:**

```yaml
workflow:
  name: My Awesome App Pipeline
  rules:
    - if: $CI_COMMIT_BRANCH == 'main'
```

Здесь `$CI_COMMIT_BRANCH` — это встроенная переменная GitLab CI/CD, которая содержит имя текущей ветки. Если коммит идёт не в `main` — пайплайн просто не запускается.

---

### 1.3 Jobs (Задания)

**Job** — это фундаментальная единица выполнения в пайплайне. Каждое задание выполняется на **GitLab Runner** (общем или собственном).

**Ключевые характеристики:**

- **Выполнение:** задание запускается на GitLab Runner (shared или self-hosted)
- **Выбор раннера:** директива `tags` позволяет выбрать конкретный раннер
- **Принадлежность к стадии:** каждое задание относится к определённой стадии через директиву `stage`

**Пример задания с выбором конкретного раннера:**

```yaml
unit_test_job:
  stage: test
  tags:
    - saas-linux-small-amd64
```

> ⚠️ **Важно:** никогда не помещайте чувствительные данные (токены, пароли, ключи) непосредственно в блок `script`. Используйте CI/CD Variables (переменные окружения GitLab) для хранения секретов.

---

### 1.4 Scripts (Скрипты)

Секция `script` — это список shell-команд, которые выполняет задание. Команды выполняются **последовательно**, одна за другой.

**Типичные сценарии использования:**
- Установка зависимостей
- Сборка кода
- Запуск тестов
- Деплой артефактов
- Сканирование безопасности

Помимо основного блока `script`, существуют два хука:

- **`before_script`** — команды настройки среды, выполняемые до основного скрипта (установка Node.js, конфигурация БД и т.д.)
- **`after_script`** — команды очистки, выполняемые после основного скрипта (удаление временных файлов и т.д.)

**Пример с тремя блоками:**

```yaml
before_script:
  - echo "Setting up environment"
script:
  - npm install
  - npm test
after_script:
  - echo "Cleaning up"
```

Важный нюанс: `after_script` выполняется **всегда** — даже если основной скрипт упал с ошибкой. Это делает его идеальным местом для операций очистки.

---

### 1.5 Stages (Стадии)

**Стадии** определяют порядок выполнения в пайплайне. Задания внутри одной стадии выполняются **параллельно**. Стадии выполняются **последовательно** — следующая стадия начинается только после успешного завершения всех заданий предыдущей.

**Типичные стадии:**

| Стадия | Назначение | Примеры команд |
|---|---|---|
| build | Компиляция или сборка кода | `mvn package`, `docker build` |
| test | Запуск автоматических тестов | `npm test`, `pytest` |
| deploy | Выкатка артефактов в окружение | `kubectl apply`, `aws s3 sync` |
| security | Сканирование на уязвимости | `trivy image`, `npm audit` |

---

### 1.6 Полный пример `.gitlab-ci.yml`

Вот пример конфигурации с двумя стадиями — `test` и `deploy` — и условием запуска:

```yaml
workflow:
  name: My Awesome App Pipeline
  rules:
    - if: $CI_COMMIT_BRANCH == 'main'

stages:
  - test
  - deploy

unit_test_job:
  stage: test
  tags:
    - saas-linux-small-amd64
  before_script:
    - echo "Install NodeJS"
  script:
    - npm install
    - npm test
  after_script:
    - echo "Tests complete"

deploy_job:
  stage: deploy
  script:
    - echo "Deploying to production..."
```

**Разбор конфигурации:**

1. `workflow.rules` — пайплайн запускается только при коммитах в `main`
2. `stages` — определяет порядок: сначала `test`, потом `deploy`
3. `unit_test_job` — принадлежит стадии `test`, запускается на Linux-раннере, выполняет установку зависимостей и тесты
4. `deploy_job` — запускается только после успешного прохождения `unit_test_job`, выполняет деплой

---

### 1.7 Типы пайплайнов

GitLab поддерживает несколько типов пайплайнов для разных сценариев:

| Тип пайплайна | Описание |
|---|---|
| Basic Pipeline | Запускает все задания в каждой стадии параллельно, затем переходит к следующей стадии последовательно |
| DAG Pipeline | Запускает задания на основе явных зависимостей для максимального параллелизма |
| Merge Request Pipeline | Валидирует новые изменения в merge requests перед слиянием |
| Merged Results Pipeline | Тестирует результат слияния MR, чтобы заранее поймать конфликты |
| Merge Train | Выстраивает MR в очередь для упорядоченной интеграции |
| Parent-Child Pipelines | Разбивает большие пайплайны на родительский и дочерние конфиги — идеально для монорепозиториев |
| Multi-Project Pipelines | Координирует пайплайны нескольких репозиториев для кросс-командных воркфлоу |

**Когда что использовать:**

- **Basic Pipeline** — подходит для большинства небольших и средних проектов с линейным воркфлоу
- **DAG Pipeline** — когда задания имеют сложные зависимости и нужно максимизировать параллелизм
- **Merge Request Pipeline** — для защиты основной ветки от некачественного кода
- **Parent-Child / Multi-Project** — для больших организаций с монорепозиториями или микросервисной архитектурой

## Часть 2. Создание первого пайплайна — пошаговое руководство

### 2.1 Шаг 1: Создание GitLab Group

Группы в GitLab служат для организации проектов. Чтобы создать группу `demos-group`:

1. Перейдите в **Groups → Create group**
2. Установите **Group name**: `demos-group` — URL группы станет `gitlab.com/demos-group`
3. Выберите **Visibility Level**: **Public**
4. Нажмите **Create group**

> **Важно:** не используйте точки в именах групп, если планируете интеграцию с SCIM или другими identity providers — это может вызвать проблемы.

После создания группы вы попадёте на её дашборд, откуда можно создавать подгруппы и проекты.

---

### 2.2 Шаг 2: Создание нового проекта

Внутри `demos-group` создадим проект `hello-world`:

1. Нажмите **New project → Create blank project**
2. Введите **Project name**: `hello-world` — URL проекта: `gitlab.com/demos-group/hello-world`
3. Установите **Visibility**: **Public**
4. Поставьте галочку **Initialize repository with a README**
5. Нажмите **Create project**

Если у вас уже есть локальный код, его можно подключить к созданному репозиторию:

```bash
git remote add origin https://gitlab.com/demos-group/hello-world.git
git branch -M main
git push -uf origin main
```

---

### 2.3 Шаг 3: Настройка CI/CD пайплайна

Перейдите в **CI/CD → Pipelines → Setup CI/CD** (или **Configure pipeline**). Откроется Pipeline Editor — встроенный редактор с поддержкой синтаксиса YAML и мгновенной валидацией конфигурации. Здесь доступны готовые шаблоны для разных языков и фреймворков.

Если оставить файл пустым или с ошибочной конфигурацией — GitLab сразу покажет сообщение об ошибке вроде: `"job configuration is missing a script or trigger keyword"`.

**Добавляем минимальный пайплайн:**

```yaml
stages:
  - build
  - test
  - deploy

first_job:
  stage: build
  script:
    - echo "This is our first GitLab CI Job"
    - ls
    - cat README.md
```

Коммитим изменение прямо в `main` — это автоматически запускает пайплайн.

---

### 2.4 Шаг 4: Просмотр результатов

После коммита перейдите в **CI/CD → Pipelines**. Вы увидите новый пайплайн со статусом **Passed** после завершения. Кликните на ID пайплайна, чтобы увидеть стадии и задания. Выберите `first_job` — откроется полный лог выполнения:

```bash
$ echo "This is our first GitLab CI Job"
This is our first GitLab CI Job
$ ls
README.md
$ cat README.md
## Hello World
## Getting started
To make it easy for you to get started with GitLab...
```

**Что происходит под капотом:** GitLab Shared Runner автоматически подготавливает Docker-контейнер, получает код из репозитория, выполняет команды из `script` и очищает среду после успешного завершения.

## Часть 3. Типы раннеров — Shared vs Self-Managed

### 3.1 Три ключевых компонента

GitLab CI/CD состоит из трёх основных компонентов:

1. **GitLab Server** — центральная платформа
2. **Runner** — агент, подбирающий и выполняющий задания CI/CD
3. **Executor** — механизм, определяющий изолированную среду выполнения

### 3.2 Сравнение Shared и Self-Managed раннеров

| Аспект | Shared Runners | Self-Managed Runners |
|---|---|---|
| Хостинг | Хостируются GitLab | На ваших серверах или в облаке |
| Настройка и обслуживание | Полностью управляется GitLab | Требует локальной установки и поддержки |
| Конфигурация | Предопределена, ограниченная кастомизация | Полностью настраиваемое окружение |
| Масштабируемость | Автоматическое масштабирование | Ручное масштабирование |
| Граница безопасности | Общая для нескольких проектов | Изолирована на проект или группу |
| Стоимость | Включена в тарифные планы GitLab | Затраты на инфраструктуру, лицензии, обслуживание |
| Идеальный сценарий | Общие задачи, небольшие команды | Строгое соответствие нормативам, кастомный инструментарий |

---

### 3.3 Shared Runners — подробнее

**Shared Runners предоставляются GitLab как сервис.** Они обслуживают задания любого проекта в GitLab SaaS по принципу "первым пришёл — первым обслужен".

**Плюсы:**
- **Нулевая настройка** — готовы к использованию сразу
- **Экономичность** — включены в большинство тарифных планов GitLab
- **Автоматическое масштабирование** — мощности подстраиваются под спрос

**Минусы:**
- **Ограниченная кастомизация** — нельзя устанавливать дополнительное ПО или менять низкоуровневые настройки
- **Конкуренция за ресурсы** — производительность может снижаться при высокой нагрузке
- **Общая граница безопасности** — все проекты используют одну и ту же среду раннера

> Shared Runners идеальны для небольших команд или open-source проектов, где простота использования важнее глубокой кастомизации.

---

### 3.4 Self-Managed Runners — подробнее

**Self-Managed Runners работают на вашей собственной инфраструктуре** — on-premises или в облаке — давая полный контроль над конфигурацией, безопасностью и производительностью.

**Плюсы:**
- **Полный контроль** — устанавливайте кастомное ПО, настраивайте производительность, управляйте обновлениями
- **Выделенные ресурсы** — сокращение времени ожидания заданий и стабильная производительность сборок
- **Изолированная среда** — минимизация взаимного влияния проектов и соответствие строгим требованиям безопасности

**Минусы:**
- **Операционная нагрузка** — требует экспертизы для установки, мониторинга и обслуживания
- **Ручное масштабирование** — нужно самостоятельно добавлять или убирать машины при изменении спроса
- **Дополнительные затраты** — железо, лицензии и текущие расходы на обслуживание

> ⚠️ При использовании self-managed раннеров в публичных или ненадёжных сетях обязательно внедрите файрволы, управление секретами и сетевые политики.

---

## Часть 4. Типы Executor'ов

### 4.1 Что такое Executor

**Executor (экзекьютор)** определяет операционную систему, инструменты и ресурсы, доступные каждому заданию. Это обеспечивает согласованность и повторяемость на разных раннерах. Выбор подходящего экзекьютора критически важен для производительности, безопасности и удобства обслуживания.

**Сравнительная таблица:**

| Executor | Лучше всего подходит для | Уровень изоляции |
|---|---|---|
| Shell | Быстрые локальные скрипты | Низкий |
| Docker | Воспроизводимые сборки | Высокий |
| Kubernetes | Масштабируемые параллельные задачи | Высокий |
| Virtual Machine | Полная изоляция на уровне ОС | Очень высокий |
| SSH | Специализированное или устаревшее железо | Средний |
| Custom | Кастомные оркестрационные воркфлоу | Варьируется |

---

### 4.2 Shell Executor

Запускает задания непосредственно на хосте раннера, используя нативный шелл (Bash или PowerShell). Идеален для быстрых скриптов или простых команд без контейнеризации.

**Плюсы:**
- Минимальная настройка и зависимости
- Нет оверхеда от контейнеров или VM
- Прямой доступ к инструментам и файловой системе хоста

**Минусы:**
- Нет изоляции — задания делят ОС и ресурсы хоста
- Результаты зависят от установленных на хосте инструментов
- Расширенный доступ создаёт риски безопасности

**Когда использовать:** для простых скриптов в доверенных средах, когда изоляция некритична.

---

### 4.3 Docker Executor

Запускает каждое задание внутри Docker-контейнера, загруженного из registry. Гарантирует чистые, воспроизводимые окружения.

**Плюсы:**
- Сильная изоляция между заданиями
- Воспроизводимые сборки через Docker-образы
- Простое управление зависимостями и кэшированием

**Минусы:**
- Требует Docker daemon и его настройки
- Оверхед контейнера на CPU и память
- Ограниченный доступ к ресурсам хоста по умолчанию

**Когда использовать:** в большинстве CI/CD сценариев — золотой стандарт для воспроизводимых сборок.

**Пример использования в `.gitlab-ci.yml`:**

```yaml
unit_test_job:
  stage: test
  image: node:18-alpine
  script:
    - npm install
    - npm test
```

Директива `image` указывает Docker-образ — раннер загрузит его и запустит задание внутри.

---

### 4.4 Kubernetes Executor

Планирует каждое задание как Pod в Kubernetes-кластере, используя его механизмы планирования, масштабирования и управления ресурсами.

**Плюсы:**
- Автоматическое масштабирование и балансировка нагрузки
- Использование существующей Kubernetes-инфраструктуры
- Политики безопасности на уровне Pod для изоляции

**Минусы:**
- Требует работающего Kubernetes-кластера
- Более сложная настройка и обслуживание
- Высокий порог входа для знания Kubernetes

**Когда использовать:** для больших команд с существующей Kubernetes-инфраструктурой, где нужна масштабируемость.

---

### 4.5 Virtual Machine Executor

Создаёт свежую VM для каждого задания, обеспечивая чистое, изолированное на уровне ОС окружение.

**Плюсы:**
- Полная изоляция ОС и чистое состояние
- Полный контроль над конфигурацией гостевой ОС

**Минусы:**
- Значительное потребление ресурсов (CPU, память, диск)
- Более медленный старт по сравнению с контейнерами
- Требует экспертизы в оркестрации VM

**Когда использовать:** когда требуется полная изоляция ОС — например, для тестирования системных компонентов.

---

### 4.6 SSH Executor

Выполняет задания на удалённой машине по SSH. Полезен для работы со специализированным железом (GPU) или интеграции с устаревшими системами.

**Плюсы:**
- Доступ к удалённому или проприетарному железу
- Не требует дополнительной настройки контейнеров или VM

**Минусы:**
- Необходимо безопасно управлять SSH-ключами
- Надёжность сети влияет на выполнение заданий
- Ограниченный мониторинг и изоляция ресурсов

> ⚠️ Регулярно ротируйте SSH-ключи и храните их в безопасности. Незащищённые учётные данные могут привести к несанкционированному доступу.

---

### 4.7 Custom Executor

Позволяет реализовать собственную логику выполнения для интеграции с любым окружением или оркестрационной системой. GitLab также предоставляет специализированные варианты, такие как Parallels и Docker Machine.

**Плюсы:**
- Полная кастомизация под уникальные воркфлоу
- Интеграция с проприетарными или нишевыми системами

**Минусы:**
- Высокие затраты на разработку и обслуживание
- Потенциальный оверхед ресурсов

**Когда использовать:** в редких случаях, когда ни один стандартный экзекьютор не подходит.

---

## Часть 5. Архитектура GitLab CI/CD — SaaS и Self-Managed

### 5.1 GitLab Server — что включает платформа

**GitLab Server** — это open-source DevOps платформа, централизующая весь жизненный цикл разработки ПО:

- Управление Git-репозиторием с гранулярным контролем доступа
- Отслеживание задач и планирование проектов
- Встроенные CI/CD пайплайны для автоматического тестирования и деплоя
- Реестры контейнеров и пакетов (Docker, Helm, npm и др.)
- Сканирование безопасности и соответствия требованиям, аудит и отчётность

GitLab доступен в двух вариантах:
- **GitLab SaaS** — хостируется GitLab.com
- **GitLab Self-Managed** — на вашей инфраструктуре

Оба варианта имеют одинаковый функциональный набор, но отличаются по контролю, кастомизации и операционной ответственности.

---

### 5.2 SaaS vs Self-Managed — детальное сравнение

#### GitLab SaaS

**Ключевые преимущества:**
- **Нулевой инфраструктурный оверхед** — никаких серверов для установки или обслуживания
- **Быстрый старт** — начните CI/CD за минуты, даже без DevOps-экспертизы
- **Эластичное масштабирование** — автоматическое масштабирование и оплата по факту использования
- **Высокая доступность** — аптайм с SLA и резервирование управляются GitLab.com
- **Автоматические обновления** — всегда последняя версия с патчами безопасности
- **Встроенная безопасность** — сканирование уязвимостей, проверки зависимостей, DAST/SAST как сервис

**Ограничения:**
- Ограниченная кастомизация платформы и поддержка плагинов
- Место хранения данных и конфиденциальность зависят от инфраструктуры GitLab.com
- Потенциальный рост затрат для больших команд или высоконагруженных пайплайнов

#### GitLab Self-Managed

**Ключевые преимущества:**
- **Полный контроль и кастомизация** — настройте раннеры, хранилище и плагины под свои нужды
- **Полное владение данными** — хостите код, метаданные и артефакты на своём железе
- **Отсутствие vendor lock-in** — свобода миграции между облаками или on-premises
- **Экономичность при масштабировании** — более предсказуемые затраты для крупных enterprise

**Ограничения:**
- Требует внутренней экспертизы для установки, обновлений и безопасности
- Инфраструктуру должна подготавливать и масштабировать ваша команда
- Вы несёте ответственность за патчинг, резервное копирование и аварийное восстановление

> ⚠️ Self-managed GitLab перекладывает всю ответственность за безопасность и обслуживание на вашу команду. Убедитесь в наличии надлежащего управления патчами и мониторинга.

---

### 5.3 Архитектура воркфлоу GitLab CI/CD — как всё работает вместе

Взаимодействие GitLab Server, Runner и Executor проходит через четыре фазы:

#### Фаза 1: Регистрация (Registration Phase)

1. Runner отправляет POST-запрос на `/runners/register` с **registration token**
2. GitLab Server валидирует запрос и возвращает **runner token**
3. Runner сохраняет этот токен для всех будущих взаимодействий

#### Фаза 2: Запрос задания (Job Request Phase)

1. Зарегистрированный Runner опрашивает `/jobs/request`, аутентифицируясь своим **runner token**
2. Сервер отвечает **job payload**, содержащим **job token** для этого конкретного выполнения

#### Фаза 3: Выполнение задания (Job Execution Phase)

1. Runner передаёт payload **Executor'у**
2. Executor использует **job token** для:
   - Клонирования репозитория
   - Загрузки зависимостей и артефактов
   - Выполнения CI/CD скрипта в изолированной среде (Docker, VM, shell)
3. Изоляция предотвращает взаимное влияние заданий и защищает хост раннера

#### Фаза 4: Отчётность (Job Reporting Phase)

1. Executor собирает логи, артефакты и код завершения
2. Отправляет POST-запрос на `/jobs/:id/trace` с **job token**
3. GitLab Server записывает результаты и обновляет UI пайплайна

**Сводная таблица токенов:**

| Фаза | Обмен токенами | Назначение |
|---|---|---|
| Регистрация | registration → runner | Аутентификация Runner'а на GitLab Server |
| Запрос задания | runner token → job token | Назначение и авторизация конкретного задания |
| Выполнение | job token | Безопасный доступ к репозиторию, артефактам, кэшу |
| Отчётность | job token | Отправка обновлений статуса и загрузка артефактов |

---

## Часть 6. Запуск заданий на Shared Runners — практика

### 6.1 Доступные типы Shared Runners

| Тип раннера | Тег в `.gitlab-ci.yml` | Статус |
|---|---|---|
| Linux | `saas-linux-medium-amd64` | Stable |
| GPU | `saas-gpu-medium` | Stable |
| Windows | `shared-windows` | Beta |
| macOS | `saas-macos-medium-m1` | Beta |

### 6.2 Конфигурация для разных платформ

```yaml
windows_job:

    - shared-windows
  script:

  tags:
    - saas-linux-medium-amd64
  script:
```

Директива `tags` гарантирует, что каждое задание подхватывается соответствующим shared runner'ом. Посмотреть все доступные shared runners можно в **Settings → CI/CD → Runners**.

## Итоговые выводы

**1. Четыре компонента — фундамент всего.** Pipeline, Stage, Job и Script — это базовые строительные блоки. Понимая их взаимодействие, вы можете построить пайплайн любой сложности.

**2. `.gitlab-ci.yml` — единственная точка правды.** Весь воркфлоу CI/CD описывается в одном файле в корне репозитория. Это делает конфигурацию версионируемой, прозрачной и воспроизводимой.

**3. Выбор раннера определяет баланс между удобством и контролем.** Shared Runners — для быстрого старта без инфраструктурных забот. Self-Managed Runners — для полного контроля, специфических требований безопасности и нестандартного окружения.

**4. Executor определяет уровень изоляции.** Docker Executor — золотой стандарт для большинства сценариев. Kubernetes Executor — для масштабируемых enterprise-решений. Shell Executor — только для простых и доверенных сред.

**5. Архитектура токенов обеспечивает безопасность.** Трёхуровневая система (registration token → runner token → job token) гарантирует, что каждый компонент системы имеет только те права, которые необходимы для выполнения своей функции.

**6. Теги — механизм маршрутизации заданий.** Правильное использование тегов позволяет точно направлять задания на нужные раннеры — Linux, Windows, macOS, GPU — обеспечивая кросс-платформенные сборки без лишней конфигурации.

**7. SaaS vs Self-Managed — не вопрос качества, а вопрос приоритетов.** SaaS подходит командам, которые хотят сосредоточиться на коде, а не на инфраструктуре. Self-Managed — для организаций с требованиями к данным, кастомному окружению или предсказуемым затратам при больших объёмах.

## Часть 1. Установка зависимостей через `before_script`

### 1.1 Зачем нужен `before_script`

В реальных пайплайнах задания часто зависят от инструментов, которых нет в базовом Docker-образе. Если попытаться использовать отсутствующий инструмент напрямую в блоке `script` — задание упадёт с ошибкой. Именно для решения этой проблемы существует `before_script`: он выполняется **перед** основным скриптом и служит местом для установки зависимостей и подготовки среды.

### 1.2 Практический пример: генерация ASCII-арта с `cowsay`

Рассмотрим конкретный сценарий — задание генерирует ASCII-арт с помощью Ruby-гема `cowsay` (утилита для отображения текста в виде речевого пузыря животного).

**Первая попытка — без установки зависимости:**

```yaml
workflow:
  name: Generate ASCII Artwork

ascii_job:
  script:
    - echo "Generating ASCII artwork using COWSAY"
    - cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" >> dragon.txt
    - grep -i "dragon" dragon.txt
    - cat dragon.txt
```

**Результат — ошибка:**

```bash
$ cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" >> dragon.txt
/usr/bin/bash: line 1: cowsay: command not found
ERROR: Job failed: exit code 1
```

Базовый Ruby-образ не содержит `cowsay`. Задание падает именно на этой строке — все последующие команды не выполняются.

### 1.3 Решение через `before_script`

```yaml
workflow:
  name: Generate ASCII Artwork

ascii_job:
  image: ruby:3.1
  before_script:
    - gem install cowsay
  script:
    - echo "Generating ASCII artwork using COWSAY"
    - cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" >> dragon.txt
    - grep -i "dragon" dragon.txt
    - cat dragon.txt
  after_script:
    - echo "Cleanup after job"
```

**Что изменилось:**

- Явно указан образ `ruby:3.1` — мы точно знаем, что используем
- В `before_script` добавлена установка гема `cowsay`
- `after_script` выполняет финальную очистку — и делает это **всегда**, даже если основной скрипт упал

**Сравнительная таблица трёх блоков:**

| Блок | Назначение | Пример |
|---|---|---|
| `before_script` | Установка зависимостей или подготовка среды перед основным скриптом | `- gem install cowsay` |
| `script` | Основные шаги задания | Команды генерации ASCII-арта |
| `after_script` | Очистка или уведомления после задания | `- echo "Cleanup after job"` |

**Успешный вывод задания:**

```bash
$ gem install cowsay
Successfully installed cowsay-0.3.0
1 gem installed

$ echo "Generating ASCII artwork using COWSAY"
Generating ASCII artwork using COWSAY

$ cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" >> dragon.txt

$ grep -i "dragon" dragon.txt
 Run for cover, I am a DRAGON....RAWR |

$ cat dragon.txt
 ________
< Run for cover, I am a DRAGON....RAWR >
 --------
        \   ^__^
         \  (oo)\_______
            (__)\       )\/\
                ||----w |
                ||     ||

$ echo "Cleanup after job"
Cleanup after job
```

> ⚠️ **Важно для воздушно-изолированных сред (air-gapped):** если у раннера нет доступа к внешним источникам пакетов — нельзя установить гем на лету. В таких случаях нужно заранее собрать кастомный Docker-образ с уже установленными зависимостями.

### 1.4 Вывод по разделу

Паттерн `before_script` + `script` + `after_script` обеспечивает предсказуемость и повторяемость пайплайна: среда всегда подготовлена перед основной работой, а очистка происходит независимо от результата. Это особенно важно в командах, где разные разработчики могут использовать разные базовые образы.

## Часть 2. Выполнение Shell-скриптов в заданиях

### 2.1 Проблема inline-команд

Когда количество команд в `script` растёт, файл `.gitlab-ci.yml` превращается в нечитаемую простыню. Представьте 50-строчный блок `script` внутри YAML — это кошмар для ревью, отладки и повторного использования.

**Пример перегруженного inline-задания:**

```yaml
workflow:
  name: Generate ASCII Artwork

ascii_job:
  before_script:
    - gem install cowsay
  script:
    - echo "Generating ASCII Artwork using COWSAY Program"
    - cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" >> dragon.txt
    - grep -i "dragon" dragon.txt
    - cat dragon.txt
  after_script:
    - echo "Executed at the end, can be used for cleaning/removing content"
```

Это ещё терпимо, но что если команд будет 30? А если логика будет повторяться в нескольких заданиях?

### 2.2 Решение: выносим логику в Shell-скрипт

**Шаг 1:** Открываем GitLab Web IDE и создаём файл `ascii-script.sh` в корне репозитория:

```bash
#!/bin/sh
echo "Generating ASCII Artwork using COWSAY Program"
cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" >> dragon.txt
grep -i "dragon" dragon.txt
cat dragon.txt
ls -ltr
```

**Шаг 2:** Сохраняем и коммитим файл — теперь он отслеживается Git.

**Шаг 3:** Обновляем `.gitlab-ci.yml` для вызова скрипта:

```yaml
workflow:
  name: Generate ASCII Artwork

ascii_job:
  before_script:
    - gem install cowsay
  script:
    - ./ascii-script.sh
  after_script:
    - echo "Executed at the end, can be used for cleaning/removing content"
```

Теперь `script` содержит одну строку. Вся логика живёт в отдельном файле, который можно тестировать локально, ревьюить отдельно и переиспользовать в других заданиях.

### 2.3 Типичная ошибка: отсутствие прав на выполнение

Коммитим изменения, запускаем пайплайн — и получаем новую ошибку:

```bash
$ gem install cowsay
Successfully installed cowsay-0.3.0
1 gem installed
$ ./ascii-script.sh
/usr/bin/bash: ./ascii-script.sh: Permission denied
ERROR: Job failed: exit code 1
```

**Причина:** файл `ascii-script.sh` не имеет флага исполняемого файла (`+x`). Git хранит права доступа, но при создании файла через Web IDE или некоторые редакторы флаг `+x` не устанавливается автоматически.

### 2.4 Решение: добавляем `chmod +x` в `before_script`

```yaml
workflow:
  name: Generate ASCII Artwork

ascii_job:
  before_script:
    - gem install cowsay
    - chmod +x ascii-script.sh
  script:
    - ./ascii-script.sh
  after_script:
    - echo "Executed at the end, can be used for cleaning/removing content"
```

Добавляем `chmod +x ascii-script.sh` в `before_script` — права устанавливаются автоматически при каждом запуске задания.

**Успешный вывод после исправления:**

```bash
$ chmod +x ascii-script.sh
$ ./ascii-script.sh
Generating ASCII Artwork using COWSAY Program
 |  Run for cover, I am a DRAGON....RAWR  |
total 32
-rwxrwxr-x 1 root root     185 Jan 25 19:32 ascii-script.sh
-rw-rw-r-- 1 root root    6181 Jan 25 19:32 README.md
-rw-rw-r-- 1 root root     270 Jan 25 19:32 .gitlab-ci.yml
-rw-r--r-- 1 root root     948 Jan 25 19:32 dragon.txt
Running after_script
Job succeeded
```

### 2.5 Преимущества выноса логики в скрипты

Подход с отдельными shell-скриптами даёт несколько важных преимуществ:

**Читаемость** — файл `.gitlab-ci.yml` остаётся чистым и понятным: видна структура пайплайна, а не детали реализации.

**Повторное использование** — один и тот же скрипт можно вызвать из нескольких заданий, не дублируя код.

**Версионирование** — скрипты хранятся в Git, изменения отслеживаются, можно делать ревью и откатываться.

**Локальная отладка** — скрипт можно запустить на своей машине до пуша, не ждя завершения пайплайна.

**Тестирование** — отдельные shell-скрипты легче покрыть unit-тестами с помощью инструментов вроде bats (Bash Automated Testing System).

## Часть 3. Ключевые слова `stage` и `stages`

### 3.1 Зачем нужна упорядоченность стадий

По умолчанию, без явного указания стадий, все задания в GitLab CI/CD запускаются **параллельно**. Это создаёт серьёзную проблему: тесты могут начаться до завершения сборки, деплой — до прохождения тестов. Результат — хаотичные падения.

`stages` решает эту проблему, устанавливая чёткий порядок: **build → test → deploy**.

### 3.2 Синтаксис `stages` и `stage`

**`stages`** — объявляет список стадий и их порядок на уровне всего пайплайна:

```yaml
stages:
  - build
  - test
  - deploy
```

**`stage`** — указывает конкретной задаче, к какой стадии она принадлежит:

```yaml
compile_code:
  stage: build
  script:
    - echo "Compiling code..."
```

**Ключевое правило:** задания внутри одной стадии выполняются **параллельно**. Стадии выполняются **последовательно** — следующая начинается только после успешного завершения всех заданий предыдущей.

### 3.3 Полный пример пайплайна с тремя стадиями

```yaml
workflow:
  name: Generate ASCII Artwork

stages:
  - build
  - test
  - deploy

build_job:
  stage: build
  before_script:
    - gem install cowsay
    - sleep 30s
  script:
    - >
      cowsay -f dragon "Run for cover,
      I am a DRAGON....RAWR" >> dragon.txt

test_job:
  stage: test
  script:
    - sleep 10s
    - grep -i "dragon" dragon.txt

deploy_job:
  stage: deploy
  script:
    - echo "Deploying..."
```

**Разбор конфигурации:**

| Стадия | Задание | Что делает |
|---|---|---|
| build | `build_job` | Устанавливает зависимости, ждёт 30 секунд и генерирует `dragon.txt` |
| test | `test_job` | Ждёт 10 секунд, затем ищет слово "dragon" в файле |
| deploy | `deploy_job` | Запускается только если тест прошёл успешно |

- `workflow.name` — задаёт понятное название пайплайна
- `sleep 30s` / `sleep 10s` — искусственные задержки для демонстрации порядка выполнения (в реальных пайплайнах заменяются реальными операциями)
- Блок `>` в YAML — это многострочная строка, позволяющая разбить длинную команду на несколько строк

### 3.4 Критическая ошибка: файлы между стадиями не передаются автоматически

Запустим пайплайн — и увидим неожиданное падение на стадии test:

**Логи `build_job`** — всё успешно:

```bash
$ gem install cowsay
Successfully installed cowsay-3.1.0
1 gem installed
$ sleep 30s
$ cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" >> dragon.txt
```

**Логи `test_job`** — падение:

```bash
$ sleep 10s
grep: dragon.txt: No such file or directory
ERROR: Job failed: exit code 1
```

**Почему это происходит?** Каждое задание в GitLab CI/CD запускается в **своём собственном изолированном окружении** на своём раннере. Файл `dragon.txt`, созданный в `build_job`, существует только в файловой системе того контейнера, где выполнялся `build_job`. Когда запускается `test_job` — он стартует в чистом контейнере без каких-либо файлов от предыдущих заданий.

**Решение — артефакты (artifacts):**

```yaml
build_job:
  stage: build
  script:
    - cowsay -f dragon "..." >> dragon.txt
  artifacts:
    paths:
      - dragon.txt
    expire_in: 1 hour
```

Добавление `artifacts` заставляет GitLab загрузить файл `dragon.txt` на сервер после завершения `build_job`, а последующие задания автоматически скачают его перед стартом.

> **Важно:** в GitLab UI есть метрики **Pipeline Duration** и **Minutes Used** — следите за ними, если работаете на плане с ограниченными compute minutes. Лишние `sleep` команды в заданиях буквально сжигают вашу квоту.

## Часть 4. Артефакты — передача данных между стадиями

### 4.1 Что такое артефакты в GitLab CI/CD

**Артефакты** — это архивы файлов и директорий, сгенерированных заданием. После завершения задания GitLab загружает их на сервер. Последующие задания в том же пайплайне автоматически скачивают их перед запуском.

Ключевые свойства артефактов:
- Загружаются на GitLab после завершения задания
- Автоматически скачиваются downstream-заданиями в том же пайплайне
- Хранятся согласно политике `expire_in`, которую вы задаёте
- Загружаются только если статус задания соответствует настройке `when`

### 4.2 Опции конфигурации артефактов

| Опция | Описание | Пример |
|---|---|---|
| `paths` | Список файлов или директорий для архивирования | `paths: [binaries/, .config]` |
| `expire_in` | Срок хранения артефактов | `expire_in: 3 days` |
| `expose_as` | Метка для отображения в Merge Request | `expose_as: 'Test Report'` |
| `name` | Кастомное имя архива | `name: "job1-artifacts-{{CI_JOB_ID}}"` |
| `when` | Условие загрузки | `when: on_failure` |

**Подробнее о значениях `when`:**
- `on_success` — загружать только при успешном завершении (значение по умолчанию)
- `on_failure` — загружать только при падении (полезно для отчётов об ошибках)
- `always` — загружать всегда, независимо от результата

### 4.3 Полный пример пайплайна с артефактами

```yaml
stages:
  - build
  - test
  - deploy

build_job:
  stage: build
  before_script:
    - gem install cowsay
    - sleep 30s
  script:
    - cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" > dragon.txt
  artifacts:
    name: "Dragon Text File"
    paths:
      - dragon.txt
    when: on_success
    expire_in: 3 days

test_job:
  stage: test
  needs: ["build_job"]
  script:
    - sleep 10s
    - grep -i "dragon" dragon.txt

deploy_job:
  stage: deploy
  needs: ["build_job"]
  script:
    - cat dragon.txt
    - echo "deploying …"
```

**Схема работы:**

1. `build_job` генерирует `dragon.txt` и загружает его как артефакт
2. `test_job` и `deploy_job` автоматически скачивают артефакт и получают файл
3. Артефакты хранятся 3 дня, после чего автоматически удаляются

## Часть 5. Пайплайн с несколькими зависимыми заданиями — классическая проблема

### 5.1 Проблема параллельного выполнения

Рассмотрим типичный начинающий пайплайн, который написан интуитивно, но работает неправильно:

```yaml
workflow:
  name: Generate ASCII Artwork

build_job_1:
  stage: build
  before_script:
    - gem install cowsay
    - sleep 30s
  script:
    - cowsay -f dragon "Run for cover, I am a DRAGON....RAWR" >> dragon.txt

test_job_2:
  stage: test
  script:
    - sleep 10s
    - grep -i "dragon" dragon.txt

deploy_job_3:
  stage: deploy
  script:
    - cat dragon.txt
    - echo "deploying ... ..."
```

**Таблица заданий:**

| Задание | Стадия | Назначение |
|---|---|---|
| `build_job_1` | build | Устанавливает `cowsay` и генерирует `dragon.txt` |
| `test_job_2` | test | Ищет "dragon" в сгенерированном файле |
| `deploy_job_3` | deploy | Выводит ASCII-арт и печатает сообщение о деплое |

### 5.2 Почему падают задания

**Ошибка в `test_job_2`:**

```bash
$ sleep 10s
grep: dragon.txt: No such file or directory
ERROR: Job failed: exit code 1
```

**Ошибка в `deploy_job_3`:**

```bash
$ cat dragon.txt
cat: dragon.txt: No such file or directory
ERROR: Job failed: exit code 1
```

**При этом `build_job_1` успешен:**

```bash
$ gem install cowsay
Successfully installed cowsay-3.0.3
1 gem installed
$ sleep 30s
$ cowsay -f dragon "Run for cover, I am a DRAGON....RAWR!" > dragon.txt
```

### 5.3 Три причины сбоев

**Причина 1: Изолированные рабочие пространства.** Каждое задание запускается на отдельном раннере с собственной файловой системой. Файл `dragon.txt`, созданный в `build_job_1`, физически существует только в контейнере этого задания.

**Причина 2: Отсутствие передачи файлов.** По умолчанию GitLab не копирует файлы между заданиями. Без явной конфигурации артефактов downstream-задания начинают с чистого листа.

**Причина 3: Порядок выполнения не гарантирован** без явных зависимостей. Использование `sleep` — это антипаттерн: негарантированное решение, зависящее от времени, а не от реального состояния системы.

### 5.4 Правильное решение

Чтобы исправить пайплайн, нужно применить два инструмента совместно:

1. **Артефакты** — чтобы передать `dragon.txt` между заданиями
2. **Ключевое слово `needs`** — чтобы явно указать зависимости

В итоге получается надёжный, детерминированный пайплайн, где каждое задание точно знает, что нужно подождать и что нужно получить от предыдущего.

## Часть 6. Ключевое слово `needs` — Directed Acyclic Graph (DAG)

### 6.1 Что такое `needs` и зачем он нужен

По умолчанию в GitLab CI/CD задания в одной стадии выполняются **параллельно**. Но что если задания внутри одной стадии имеют зависимости между собой? Например: нельзя тестировать образ, который ещё не собран, и нельзя пушить образ, тестирование которого провалилось.

`needs` позволяет явно указать, какие задания должны завершиться **перед** стартом текущего. Это превращает линейный пайплайн в **DAG (Directed Acyclic Graph)** — направленный ациклический граф, где каждое задание запускается ровно тогда, когда готовы все его зависимости.

### 6.2 Исходный пайплайн без `needs`

```yaml
workflow:
  name: Generate ASCII Artwork

stages:
  - build
  - test
  - docker
  - deploy

build_file:
  stage: build
  before_script:
    - gem install cowsay
  script:
    - >
      cowsay -f dragon "Run for cover,
      I am a DRAGON....RAWR" >> dragon.txt
  artifacts:
    name: Dragon Text File
    paths:
      - dragon.txt
    when: on_success
    expire_in: 3 days

test_file:
  stage: test
  script:
    - echo "Running tests..."

docker_build:
  stage: docker
  script:
    - echo "docker build -t docker.io/dockerUsername/imageName:version"
    - sleep 15s

docker_testing:
  stage: docker
  script:
    - echo "docker run -p 80:80 docker.io/dockerUsername/imageName:version"
    - sleep 10s
    - exit 1

docker_push:
  stage: docker
  script:
    - echo "docker login --username=dockerUsername --password=s3cUrePaSsW0rd"
    - echo "docker push docker.io/dockerUsername/imageName:version"

deploy_ec2:
  stage: deploy
  script:
    - echo "Deploying to AWS EC2"
```

**Таблица Docker-заданий:**

| Задание | Стадия | Назначение |
|---|---|---|
| `docker_build` | docker | Сборка Docker-образа |
| `docker_testing` | docker | Запуск контейнера и проверка работоспособности |
| `docker_push` | docker | Авторизация и пуш образа в registry |

**Проблема:** все три Docker-задания принадлежат одной стадии `docker`, поэтому GitLab запускает их **параллельно**. Это катастрофа:

- `docker_push` может начать пушить образ, который ещё не собран
- `docker_testing` может упасть, но `docker_push` всё равно выполнится
- `docker_testing` намеренно завершается с `exit 1` — это симулирует проваленный тест, но `docker_push` этого не знает и продолжает работу

### 6.3 Решение: добавляем `needs`

```yaml
docker_build:
  stage: docker
  needs:
    - test_file
  script:
    - echo "docker build -t docker.io/dockerUsername/imageName:version"
    - sleep 15s

docker_testing:
  stage: docker
  needs:
    - docker_build
  script:
    - echo "docker run -p 80:80 docker.io/dockerUsername/imageName:version"
    - sleep 10s
    - exit 1

docker_push:
  stage: docker
  needs:
    - docker_testing
  script:
    - echo "docker login --username=dockerUsername --password=s3cUrePaSsW0rd"
    - echo "docker push docker.io/dockerUsername/imageName:version"
```

**Новый порядок выполнения:**

```
build_file → test_file → docker_build → docker_testing → docker_push
```

Теперь:
- `docker_build` запускается только после `test_file`
- `docker_testing` запускается только после `docker_build`
- `docker_push` запускается только после `docker_testing`
- Если `docker_testing` падает (а оно падает, `exit 1`) — `docker_push` **автоматически пропускается**

**Логи упавшего `docker_testing`:**

```bash
$ echo "docker run -p 80:80 docker.io/dockerUsername/imageName:version"
docker run -p 80:80 docker.io/dockerUsername/imageName:version
$ sleep 10s
$ exit 1
ERROR: Job failed: exit code 1
```

`docker_push` при этом не запускается вообще — именно такое поведение нам нужно.

### 6.4 Продвинутое использование `needs`: игнорирование порядка стадий

`needs` может делать не только последовательное выполнение внутри стадии. Он позволяет запустить задание **немедленно** после завершения его зависимостей — даже если это задание находится в более поздней стадии, чем текущая.

**Пример:**

```yaml
docker_build:
  stage: docker
  needs:
    - build_file
  script:
    - echo "docker build -t docker.io/dockerUsername/imageName:version"
```

Здесь `docker_build` принадлежит стадии `docker` (третья стадия), но с `needs: [build_file]` он запустится **сразу после `build_file`** — параллельно с `test_file`, не дожидаясь завершения стадии `test`. Это называется "преодоление порядка стадий" (stage order override) и позволяет максимизировать параллелизм.

**Визуализация DAG-пайплайна:**

```
build_file ──┬──► test_file ──► deploy_ec2
             │
             └──► docker_build ──► docker_testing ──► docker_push
```

Обе ветки работают параллельно, что существенно сокращает общее время пайплайна.

## Итоговые выводы

**1. `before_script` — место для подготовки среды, не для бизнес-логики.** Используйте его для установки зависимостей, настройки конфигурационных файлов, авторизации в реестрах. Основная работа — в `script`. Очистка — в `after_script`.

**2. Shell-скрипты — лучше inline-команд для сложной логики.** Как только в блоке `script` появляется больше 5-7 строк — выносите логику в отдельный файл. Помните о `chmod +x`, иначе получите `Permission denied`.

**3. Без `stages` — хаос, без `artifacts` — изоляция.** Стадии дают порядок выполнения. Артефакты решают проблему изолированных файловых систем. Оба инструмента нужны вместе для надёжной передачи данных между заданиями.

**4. `needs` — ключ к гибким, эффективным пайплайнам.** Он решает сразу две задачи: устанавливает зависимости внутри одной стадии (чего нельзя сделать через `stages`) и позволяет запускать задания раньше, не дожидаясь завершения всей предыдущей стадии.

**5. `sleep` — антипаттерн.** Использование фиксированных задержек для синхронизации — признак неправильной архитектуры пайплайна. Правильное решение — артефакты и `needs`.

**6. Каждое задание — чистый лист.** Всегда помните: задания не делят файловую систему. Если нужно передать файл — используйте артефакты. Если нужно передать данные — используйте артефакты или переменные окружения.

**7. DAG-пайплайны быстрее линейных.** Комбинация `stages` + `needs` позволяет выстраивать граф зависимостей, где независимые ветки работают параллельно. Это может сократить время пайплайна в несколько раз по сравнению с чисто последовательным выполнением.

## Часть 1. Переменные в GitLab CI/CD — уровни и области видимости

### 1.1 Проблема: жёстко закодированные значения

Рассмотрим типичную ситуацию — пайплайн с Docker-заданиями, где одни и те же значения дублируются в каждом задании:

```yaml
docker_build:
  stage: docker
  script:
    - echo "docker build -t docker.io/dockerUsername/imageName:version"

docker_testing:
  stage: docker
  script:
    - echo "docker run -p 80:80 docker.io/dockerUsername/imageName:version"

docker_push:
  stage: docker
  script:
    - echo "docker login --username=dockerUsername --password=s3cUrePaSsW0rd"
    - echo "docker push docker.io/dockerUsername/imageName:version"
```

Здесь `docker.io/dockerUsername/imageName:version` встречается в трёх местах. Стоит поменять имя образа или версию — придётся вносить правки в каждое задание. Это нарушает принцип **DRY** (Don't Repeat Yourself — не повторяйся) и создаёт риск рассинхронизации при обновлениях.

---

### 1.2 Два уровня переменных

GitLab CI/CD поддерживает переменные двух областей видимости:

| Область | Место объявления | Доступность |
|---|---|---|
| **Global (глобальные)** | Блок `variables:` на верхнем уровне | Во **всех** заданиях |
| **Job (уровень задания)** | `variables:` внутри конкретного задания | Только в этом задании |

Важное правило: **глобальные переменные могут быть переопределены** переменными уровня задания с тем же именем. Переменные уровня задания изолированы и не видны другим заданиям.

---

### 1.3 Объявление глобальных переменных

```yaml
variables:
  DEPLOY_SITE: "https://example.com/"

deploy_job:
  stage: deploy
  script:
    - deploy-script --url "$DEPLOY_SITE" --path "/"
  environment: production

deploy_review_job:
  stage: deploy
  variables:
    REVIEW_PATH: "/review"
  script:
    - deploy-review-script --url "$DEPLOY_SITE" --path "$REVIEW_PATH"
  environment: production
```

Здесь:
- `DEPLOY_SITE` — глобальная переменная, доступна в обоих заданиях
- `REVIEW_PATH` — переменная уровня задания, видна только в `deploy_review_job`

---

### 1.4 Рефакторинг Docker-пайплайна: промежуточный вариант

Первый шаг — вынести повторяющиеся значения в переменные на уровне каждого задания. Это уже лучше, чем жёсткое кодирование:

```yaml
docker_build:
  stage: docker
  variables:
    USERNAME: dockerUsername
    REGISTRY: docker.io/$USERNAME
    IMAGE: ascii-artwork
    VERSION: latest
  script:
    - echo "docker build -t $REGISTRY/$IMAGE:$VERSION"

docker_testing:
  stage: docker
  variables:
    USERNAME: dockerUsername
    REGISTRY: docker.io/$USERNAME
    IMAGE: ascii-artwork
    VERSION: latest
  script:
    - echo "docker run -p 80:80 $REGISTRY/$IMAGE:$VERSION"

docker_push:
  stage: docker
  variables:
    USERNAME: dockerUsername
    REGISTRY: docker.io/$USERNAME
    IMAGE: ascii-artwork
    VERSION: latest
    PASSWORD: s3cUrePaSsW0rd
  script:
    - echo "docker login --username=$USERNAME --password=$PASSWORD"
    - echo "docker push $REGISTRY/$IMAGE:$VERSION"
```

Но мы всё ещё повторяем `USERNAME`, `REGISTRY`, `IMAGE` и `VERSION` в каждом задании. Продолжаем рефакторинг.

---

### 1.5 Итоговый рефакторинг: глобальные переменные

Выносим все общие переменные на верхний уровень. Только чувствительные или специфичные для задания переменные остаются внутри задания:

```yaml
workflow:
  name: Generate ASCII Artwork

stages:
  - build
  - test
  - docker
  - deploy

variables:
  USERNAME: dockerUsername
  REGISTRY: docker.io/$USERNAME
  IMAGE: ascii-artwork
  VERSION: latest

build_file:
  stage: build
  script: …

test_file:
  stage: test
  script: …

docker_build:
  stage: docker
  needs:
    - build_file
  script:
    - echo "docker build -t $REGISTRY/$IMAGE:$VERSION"

docker_testing:
  stage: docker
  needs:
    - docker_build
  script:
    - echo "docker run -p 80:80 $REGISTRY/$IMAGE:$VERSION"

docker_push:
  stage: docker
  needs:
    - docker_testing
  variables:
    PASSWORD: s3cUrePaSsW0rd
  script:
    - echo "docker login --username=$USERNAME --password=$PASSWORD"
    - echo "docker push $REGISTRY/$IMAGE:$VERSION"
```

Теперь `USERNAME`, `REGISTRY`, `IMAGE` и `VERSION` объявлены ровно один раз. Только `PASSWORD` остаётся в `docker_push` — это чувствительный секрет.

---

### 1.6 Использование предопределённых переменных для динамической версионизации

Статический тег `latest` — плохая практика: образы перезаписывают друг друга и нет возможности отследить, какой именно пайплайн создал конкретный образ. Вместо этого используем встроенную переменную `$CI_PIPELINE_ID`:

```yaml
variables:
  USERNAME: dockerUsername
  REGISTRY: docker.io/$USERNAME
  IMAGE: ascii-artwork
  VERSION: $CI_PIPELINE_ID
```

Теперь каждый запуск пайплайна создаёт образ с уникальным тегом вроде `ascii-artwork:1153576211`. Это делает версии образов трассируемыми — всегда можно найти, какому коммиту соответствует образ.

**Вывод в логах после такой настройки:**

```bash
$ echo "docker push $REGISTRY/$IMAGE:$VERSION"
docker push docker.io/dockerUsername/ascii-artwork:1153576211
```

---

### 1.7 Изоляция переменных уровня задания

Критически важно понимать: переменная, объявленная в одном задании, **недоступна** в другом. Демонстрация:

```yaml
deploy_ec2:
  stage: deploy
  script:
    - echo "Username: $USERNAME, Password: $PASSWORD"
```

**Вывод в логах:**

```bash
$ echo "Username: $USERNAME, Password: $PASSWORD"
Username: dockerUsername, Password:
```

`$USERNAME` пришёл из глобальных переменных — всё верно. `$PASSWORD` пуст, потому что он был объявлен только внутри `docker_push`. Это именно то поведение, которое нам нужно с точки зрения безопасности.

> ⚠️ **Важно:** избегайте вывода чувствительных переменных в логах заданий. Используйте Masked Variables или Protected Variables для защиты учётных данных и секретов.

## Часть 2. Маскирование переменных через настройки проекта

### 2.1 Зачем маскировать переменные

Жёсткое кодирование учётных данных в `.gitlab-ci.yml` ведёт к утечкам, если репозиторий публичный или кто-то получит доступ на чтение. Маскированные переменные позволяют:

- Хранить секреты за пределами кодовой базы
- Предотвращать появление значений в выводе пайплайна
- Централизованно управлять учётными данными

**Проблемный код — пароль прямо в YAML:**

```yaml
docker_push:
  stage: docker
  variables:
    PASSWORD: s3cUrePaSsW0rd
  script:
    - echo "docker login --username=$USERNAME --password=$PASSWORD"
```

Такой подход выставляет пароль в репозитории и в логах задания.

---

### 2.2 Создание маскированной переменной в настройках проекта

Путь: **Settings → CI/CD → Variables → Add variable**

| Параметр | Значение |
|---|---|
| **Key** | `DOCKER_PASSWORD` |
| **Value** | `s3cUrePaSsW0rd` |
| **Masked** | Скрывает значение в логах заданий |
| **Protected** | Только на защищённых ветках (опционально) |
| **Environment scope** | `*` — доступна во всех окружениях |

После сохранения переменная появляется в списке с флагом **masked**.

> ⚠️ **Внимание:** любой пользователь с правами Developer или Maintainer может просмотреть или изменить значение переменной в настройках проекта.

---

### 2.3 Использование маскированной переменной в пайплайне

После создания переменной в настройках проекта больше не нужно объявлять `PASSWORD` в задании:

```yaml
docker_push:
  stage: docker
  needs:
    - docker_testing
  script:
    - echo "docker login --username=$USERNAME --password=$DOCKER_PASSWORD"
    - echo "docker push $REGISTRY/$IMAGE:$VERSION"
```

Глобальная переменная `$DOCKER_PASSWORD` автоматически внедряется в окружение задания из настроек проекта.

**Как это выглядит в логах:**

```bash
$ echo "docker login --username=dockerUsername --password=[MASKED]"
docker login --username=dockerUsername --password='[MASKED]'
```

Пароль заменяется на `[MASKED]` — даже если кто-то откроет логи, он не увидит реальное значение.

---

### 2.4 Скрытие заданий для демонстрации

Интересный приём: чтобы запустить только часть заданий в целях демонстрации или тестирования, добавьте точку (`.`) перед именем задания. Такие задания отображаются в редакторе, но **пропускаются при выполнении**:

```yaml
stages:
  - build
  - test
  - docker
  - deploy

variables:
  USERNAME: dockerUsername
  REGISTRY: docker.io/$USERNAME
  IMAGE: imageName
  VERSION: $CI_PIPELINE_ID

.build_file:
  script: [...]

.test_file:

.docker_build:

.docker_testing:

docker_push:
  stage: docker
  script:
    - echo "docker login --username=$USERNAME --password=$DOCKER_PASSWORD"
    - echo "docker push $REGISTRY/$IMAGE:$VERSION"

deploy_ec2:
  stage: deploy
  script:
    - echo "deploying ..."
    - echo "Username - $USERNAME and Password - $DOCKER_PASSWORD"
```

Задания `.build_file`, `.test_file` и т.д. не выполняются. Запустятся только `docker_push` и `deploy_ec2`. Это удобно для отладки конкретных заданий без прогона всего пайплайна.

## Часть 3. Предопределённые переменные GitLab CI/CD

### 3.1 Что такое предопределённые переменные

GitLab автоматически предоставляет десятки переменных окружения для каждого задания. Они содержат контекстную информацию о заданиях, пайплайнах, репозиториях и пользователях — без какой-либо настройки с вашей стороны.

**Категории предопределённых переменных:**

| Категория | Доступна в | Сценарий использования |

| Pipeline-level Variables | Конфиг пайплайна и скрипты заданий | Определить источник пайплайна, ID, статус |
| Runner-level Variables | Выполнение заданий | Получить данные конкретного раннера |
| Merge Request Variables | Пайплайны merge request | Получить метки MR, ветки, заголовок |

## Часть 4. Merge Requests — концепция и воркфлоу

### 4.1 Что такое Merge Request

**Merge Request (MR)** — это механизм предложения, ревью и интеграции изменений кода из одной ветки в другую. MR позволяют:

- Предлагать изменения из **исходной ветки** в **целевую** (обычно `main`)
- Сотрудничать через комментарии, обсуждения кода и одобрения
- Применять контроль качества, требуя прохождения CI/CD пайплайнов перед слиянием

**Сравнение GitLab MR и GitHub PR:**

| Концепция | GitLab Merge Request | GitHub Pull Request |
|---|---|---|
| Название функции | Merge Request | Pull Request |
| Целевая ветка | `main` (настраивается) | `main` или `master` |
| Интеграция CI/CD | Встроенный GitLab CI/CD | Внешний или GitHub Actions |
| Черновой режим | Work in Progress MR | Draft Pull Request |

---

### 4.2 Типичный воркфлоу Merge Request

**Шаг 1: Создание feature-ветки**

```bash
git checkout -b feature-xyz
```

**Шаг 2: Разработка и коммит изменений**

```bash
# Редактируем файлы...
git add .
git commit -m "Implement feature XYZ"
```

**Шаг 3: Пуш в удалённый репозиторий**

```bash
git push -u origin feature-xyz
```

**Шаг 4: Открытие Merge Request** через GitLab UI: **Merge Requests → New merge request**, выбрать `feature-xyz` как source branch и `main` как target branch.

**Шаг 5: Ревью и итерации** — рецензенты оставляют комментарии, разработчик пушит исправления:

```bash
git add .
git commit -m "Address review feedback"
git push
```

**Шаг 6: Слияние** — после всех одобрений и успешного прохождения CI нажимаем **Merge**.

> ⚠️ **Не выполняйте слияние, если пайплайн провалился.** Упавшие тесты или ошибки линтера попадут в основную ветку.

## Часть 5. Ключевое слово `rules` на уровне задания

### 5.1 Зачем нужны `rules` на уровне задания

Иногда нужно, чтобы конкретное задание выполнялось только при определённых условиях. Например, задание с переменными merge request не имеет смысла запускать при обычном пуше — там нет данных для вывода. Директива `rules` позволяет точно контролировать это.

---

### 5.2 Применение `rules` для ограничения задания

Добавляем условие, которое проверяет источник пайплайна:

```yaml
merge_request_predefined_variables:
  stage: test
  rules:
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
  script:
    - echo "CI_MERGE_REQUEST_LABELS = $CI_MERGE_REQUEST_LABELS"
    - echo "CI_MERGE_REQUEST_TARGET_BRANCH_NAME = $CI_MERGE_REQUEST_TARGET_BRANCH_NAME"
    - echo "CI_MERGE_REQUEST_ASSIGNEES = $CI_MERGE_REQUEST_ASSIGNEES"
    - echo "CI_MERGE_REQUEST_SOURCE_BRANCH_NAME = $CI_MERGE_REQUEST_SOURCE_BRANCH_NAME"
    - echo "CI_MERGE_REQUEST_TITLE = $CI_MERGE_REQUEST_TITLE"
```

Теперь задание `merge_request_predefined_variables` включается **только** когда `CI_PIPELINE_SOURCE` равен `merge_request_event`.

> ⚠️ **Важно:** всегда оборачивайте условное выражение в одинарные кавычки, чтобы избежать ошибок парсинга YAML.

---

### 5.3 Практическая демонстрация

**Шаг 1:** Создаём feature-ветку и пушим изменения:

```bash
git checkout -b feature-1
git push -u origin feature-1
```

Результат: пайплайн запускается, но задание `merge_request_predefined_variables` **не выполняется** — это был обычный пуш, не MR-событие. Работает только `generic_predefined_variables`.

**Шаг 2:** Создаём MR в GitLab UI, добавляем метки `predefined-variables` и `testing-rules`.

**Шаг 3:** GitLab запускает MR-пайплайн. Теперь задание `merge_request_predefined_variables` **включается** — источник пайплайна стал `merge_request_event`.

**Реальный вывод задания в MR-пайплайне:**

```bash
CI_MERGE_REQUEST_LABELS - predefined-variables,testing-rules
CI_MERGE_REQUEST_TARGET_BRANCH_NAME - main
CI_MERGE_REQUEST_ASSIGNEES - sidd-harth
CI_MERGE_REQUEST_SOURCE_BRANCH_NAME - feature-1
CI_MERGE_REQUEST_TITLE - added rules for merge request job
```

Все переменные теперь заполнены — потому что пайплайн запущен именно событием merge request.

## Часть 6. `rules` на уровне `workflow` — управление всем пайплайном

### 6.1 Разница между job-level и workflow-level rules

- **Job-level rules** — определяют, запускается ли **конкретное задание**
- **Workflow-level rules** — определяют, **создаётся ли вообще пайплайн**

Если workflow-level rule не выполняется — пайплайн не создаётся вообще, ни одно задание не запустится. Это позволяет экономить ресурсы на заведомо ненужных пайплайнах.

---

### 6.2 Правило 1: запуск только на `main`

```yaml
workflow:
  name: Exploring GitLab CI Concepts
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      variables:
        DEPLOY_VARIABLE: "PRODUCTION"

deploy-job:
  script:
    - echo "Deploying application..."
    - echo "Application successfully deployed to $DEPLOY_VARIABLE environment"
```

Пайплайн создаётся только при пуше в `main`. При этом переменная `DEPLOY_VARIABLE` автоматически устанавливается в `PRODUCTION`. Пуши в другие ветки не запускают пайплайн вообще.

**Вывод при пуше в `main`:**

```bash
$ echo "Application successfully deployed to $DEPLOY_VARIABLE environment"
Application successfully deployed to PRODUCTION environment
```

---

### 6.3 Правило 2: MR из feature-веток

```yaml
workflow:
  name: Exploring GitLab CI Concepts
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      variables:
        DEPLOY_VARIABLE: "PRODUCTION"
    - if: '$CI_MERGE_REQUEST_SOURCE_BRANCH_NAME =~ /^feature\// && $CI_PIPELINE_SOURCE == "merge_request_event"'
      variables:
        DEPLOY_VARIABLE: "TESTING"
```

Здесь используется **регулярное выражение**: `/^feature\//` — исходная ветка MR должна начинаться с `feature/`. Одновременно проверяется, что источник — именно merge request. При совпадении устанавливается `DEPLOY_VARIABLE: "TESTING"`.

**Примеры веток:**
- `feature/my-awesome-feature` — подходит, пайплайн создаётся
- `bugfix/fix-login` — не подходит, пайплайн не создаётся
- `main` — подходит под первое правило

---

### 6.4 Правило 3: фильтрация по изменениям файлов

```yaml
workflow:
  name: Exploring GitLab CI Concepts
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      variables:
        DEPLOY_VARIABLE: "PRODUCTION"
    - if: '$CI_MERGE_REQUEST_SOURCE_BRANCH_NAME =~ /^feature\// && $CI_PIPELINE_SOURCE == "merge_request_event"'
      changes:
        - README.md
      variables:
        DEPLOY_VARIABLE: "TESTING"
```

Директива `changes:` добавляет дополнительное условие: пайплайн создаётся только если в MR изменён файл `README.md`. Если в MR изменены другие файлы — правило не срабатывает. Это мощный механизм для экономии ресурсов: не нужно гонять тесты, если изменилась только документация.

**Вывод при создании MR из `feature/something` с изменением `README.md`:**

```bash
$ echo "Application successfully deployed to $DEPLOY_VARIABLE environment"
Application successfully deployed to TESTING environment
```

---

### 6.5 Сводная таблица правил

| Правило | Триггер | Переменная | Описание |
|---|---|---|---|
| 1 | `CI_COMMIT_BRANCH == "main"` | PRODUCTION | Сборка только на `main` |
| 2 | MR-событие + ветка начинается с `feature/` | TESTING | Для MR из feature-веток |
| 3 | То же, что правило 2 + изменён `README.md` | TESTING | Только при изменении README |

> ⚠️ **Критическое правило:** срабатывает **первое** совпадающее правило. Если ни одно правило не совпадает — пайплайн не создаётся.

---

### 6.6 Финальный пример с продвинутыми правилами

```yaml
workflow:
  name: Exploring GitLab CI Concepts
  rules:
    - if: '$CI_COMMIT_TITLE =~ /-draft$/'
      when: never
    - if: '$CI_PIPELINE_SOURCE == "merge_request_event"'
    - if: '$CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH'
```

Разбор трёх правил:

1. Если заголовок коммита заканчивается на `-draft` — пайплайн **никогда** не запускается (`when: never`)
2. При любом merge request — пайплайн запускается
3. При пуше в дефолтную ветку (обычно `main`) — пайплайн запускается

`$CI_DEFAULT_BRANCH` — предопределённая переменная, которая автоматически содержит имя дефолтной ветки проекта. Это лучше, чем хардкодить `"main"`, т.к. у некоторых проектов дефолтная ветка называется иначе.

## Часть 7. Расписания пайплайнов (Pipeline Schedules)

### 7.1 Зачем нужны расписания

Иногда пайплайн нужно запускать не при каждом пуше, а по расписанию: ежечасно, ежедневно или по произвольному cron-выражению. Типичные сценарии:

- Еженощные деплои в определённое время
- Запуск обслуживающих скриптов без ручного вмешательства
- Еженедельные отчёты о состоянии зависимостей
- Синхронизация данных по расписанию

### 7.2 Настройка расписания

Путь: **CI/CD → Schedules → New schedule**

| Параметр | Описание | Пример |
|---|---|---|
| Interval | Пресет или cron-выражение | `0 0 1 1 *` |
| Timezone | Часовой пояс региона | `Asia/Kolkata (Mumbai)` |
| Target branch | Ветка или тег для расписания | `main` |
| CI/CD variables | Пары ключ-значение для этого запуска | `DEPLOY_VARIABLE=manual deployment` |

**Формат cron-выражения:**

```
minute hour day-of-month month day-of-week
```

Примеры:
- `0 0 1 1 *` — раз в год, в полночь 1 января
- `0 9 * * 1-5` — каждый будний день в 9:00
- `*/15 * * * *` — каждые 15 минут

### 7.3 Пайплайн с переменной из расписания

```yaml
deploy-job:
  stage: deploy
  script:
    - echo "Deploying application..."
    - echo "Application successfully deployed to $DEPLOY_VARIABLE environment"
```

При настройке расписания добавляем переменную:

| Ключ | Значение |
|---|---|
| DEPLOY_VARIABLE | manual deployment |

**Вывод при запуске по расписанию:**

```bash
$ echo "Application successfully deployed to $DEPLOY_VARIABLE environment"
Application successfully deployed to manual deployment environment
```

### 7.4 Управление расписаниями

- Все расписания отображаются с деталями: следующий запуск, cron-выражение, целевая ветка
- Кнопка **Play** позволяет запустить расписание немедленно, не дожидаясь следующего срабатывания
- Пайплайны, запущенные по расписанию, помечаются тегом **scheduled** в списке пайплайнов

> ⚠️ Имена переменных должны точно совпадать с тем, что используется в `.gitlab-ci.yml`. Опечатка приведёт к падению задания.

## Часть 8. Resource Groups — взаимное исключение заданий

### 8.1 Проблема параллельных деплоев

По умолчанию пайплайны в GitLab выполняются **параллельно**. Если два разработчика одновременно запустили пайплайны, которые деплоят в production — оба деплоя начнутся одновременно. Это может привести к:

- Гонке состояний при деплое
- Непредсказуемому результату из-за переплетения операций
- Откату одного деплоя поверх другого

**Resource Groups** решают эту проблему, обеспечивая **взаимное исключение**: только одно задание в группе может работать одновременно. Остальные стоят в очереди.

### 8.2 Применение `resource_group`

```yaml
deploy-to-production:
  stage: deploy
  script:
    - deploy
  resource_group: production
```

Теперь если несколько пайплайнов одновременно дойдут до `deploy-to-production` — только один получит ресурс `production` и начнёт работу. Остальные ждут в очереди.

### 8.3 Режимы процесса (Process Modes)

Resource groups поддерживают три режима обработки очереди:

| Режим | Описание |
|---|---|
| `unordered` | По умолчанию; любое ожидающее задание может начать следующим (без гарантий) |
| `oldest_first` | Задания выполняются в порядке постановки в очередь |
| `newest_first` | Первым выполняется последнее поставленное в очередь |

Изменить режим можно через GitLab CI REST API.

**Пример для понимания режимов.** Запустили три пайплайна почти одновременно:

- **unordered**: любой из deploy-1, deploy-2, deploy-3 может запуститься первым
- **oldest_first**: deploy-1 → deploy-2 → deploy-3 (наиболее предсказуемо)
- **newest_first**: deploy-3 → deploy-2 → deploy-1 (для сценариев "всегда деплоим свежий код")

### 8.4 Демонстрация с `sleep`

```yaml
deploy-job:
  stage: deploy
  resource_group: production
  script:
    - echo "Deploying application..."
    - sleep 300
    - echo "Application successfully deployed"
```

1. Делаем коммит в `main` — запускается первый пайплайн, `deploy-job` начинает работу и "занят" 5 минут
2. Запускаем второй пайплайн вручную — его `deploy-job` видит: ресурс `production` занят, и **ждёт**
3. GitLab UI показывает второй `deploy-job` со статусом "waiting for resource `production`"
4. Отменяем первый пайплайн — ресурс освобождается — второй `deploy-job` **автоматически** начинает работу

## Часть 9. Таймаут заданий (Job Timeout)

### 9.1 Зачем нужен таймаут

Зависшие или слишком долго выполняющиеся задания — реальная проблема:

- Блокируют раннеры, не давая другим заданиям запуститься
- Потребляют ресурсы впустую
- Создают сюрпризы в счётчике compute minutes
- Могут быть симптомом бесконечного цикла или зависшей сетевой операции

По умолчанию GitLab устанавливает максимальный таймаут на уровне проекта, но можно задать его и для каждого задания индивидуально.

### 9.2 Добавление таймаута в задание

Директива `timeout` поддерживает человекочитаемые форматы длительности:

```yaml
build:
  script: ./build.sh
  timeout: 3 hours 30 minutes

test:
  script: rspec
  timeout: 3h 30m

lint:
  script: ./lint.sh
  timeout: 3600 seconds
```

**Поддерживаемые форматы длительности:**

| Формат | Примеры |
|---|---|
| Часы и минуты | `3 hours 30 minutes` |
| Компактный (h/m/s) | `3h30m`, `45m`, `10s` |
| Полные слова | `1 hour`, `90 seconds` |
| Комбинированные | `1h 15m`, `2h5m30s` |

### 9.3 Принудительное быстрое завершение

Пример задания, которое будет отменено через 10 секунд:

```yaml
deploy-job:
  resource_group: production
  timeout: 10s
  script:
    - echo "Deploying application..."
    - sleep 300s
    - echo "Application successfully deployed"
```

Когда задание превышает 10 секунд, раннер завершает его принудительно:

```
ERROR: Job failed: execution took longer than 10 seconds
```

> ⚠️ **Совет:** не ставьте таймаут слишком коротким — это приведёт к неожиданным падениям в нормальных условиях. Выбирайте значение, которое учитывает обычное время выполнения плюс запас.

## Часть 10. Docker-образы для заданий (`image:`)

### 10.1 Проблема: инструмент недоступен в базовом образе

Если задание не указывает образ явно, раннер использует образ по умолчанию (часто `ruby:3.1`). Node.js в нём недоступен:

```bash
Using Docker executor with image ruby:3.1 ...
$ node -v
/usr/bin/bash: line 140: node: command not found
ERROR: Job failed: exit code 1
```

Существует два способа решить эту проблему.

**Сравнение подходов:**

| Подход | Плюсы | Минусы |
|---|---|---|
| Установка в `before_script` | Работает с любым базовым образом | Добавляет оверхед, дублирует работу на каждом раннере |
| Использование `image:` | Быстро, воспроизводимо, минимальная настройка | Нужно выбрать или собрать подходящий образ |

### 10.2 Установка Node.js в `before_script` (допустимо, но медленно)

```yaml
deploy-job:
  before_script:
    - |
      apt-get update && apt-get install -y ca-certificates curl gnupg
      curl -fsSL https://deb.nodesource.com/gpgkey/nodesource-repo.gpg.key | gpg --dearmor -o /etc/apt/keyrings/nodesource.gpg
      NODE_MAJOR=20
      echo "deb [signed-by=/etc/apt/keyrings/nodesource.gpg] https://deb.nodesource.com/node_$NODE_MAJOR.x nodistro main" | tee /etc/apt/sources.list.d/nodesource.list
      apt-get update && apt-get install -y nodejs
  script:
    - node -v
    - npm -v
```

Обратите внимание на синтаксис `|` (YAML literal block) — он позволяет писать многострочный скрипт читаемо, как один блок кода.

**Вывод после успешной установки:**

```bash
$ node -v
v20.11.0
$ npm -v
10.2.4
Job succeeded
```

### 10.3 Использование `image:` — рекомендуемый подход

```yaml
deploy-job:
  resource_group: production
  image: node:20-alpine3.18
  script:
    - node -v
    - npm -v
```

Раннер загружает образ `node:20-alpine3.18`, в котором Node.js уже установлен. Никакой дополнительной установки не требуется.

**Вывод:**

```bash
Using Docker executor with image node:20-alpine3.18 ...
$ node -v
v20.11.0
$ npm -v
10.2.4
Job succeeded
```

Результат тот же, но время выполнения значительно меньше — нет шага установки Node.js.

### 10.4 Глобальный образ через `default:`

Если несколько заданий используют один образ — устанавливаем его глобально:

```yaml
default:
  image: node:20-alpine3.18

job1:
  script:
    - node -v

job2:
  script:
    - npm -v
```

Оба задания используют `node:20-alpine3.18` без необходимости повторять `image:` в каждом.

**Практические советы:**
- Используйте официальные образы (например, `node:<version>-alpine` для меньшего размера)
- Если проекту нужны дополнительные пакеты — создайте кастомный образ с Dockerfile и опубликуйте его в registry
- `image:` можно указывать как на уровне задания, так и глобально через `default:` — в зависимости от потребностей пайплайна

## Часть 11. Parallel Matrix — параллельные задания

### 11.1 Разделение тестов с `parallel`

Если у вас большой тестовый набор — например, 100 RSpec-тестов, которые занимают 20 минут — можно разбить их на несколько параллельных фрагментов:

```yaml
test:
  stage: test
  image: ruby:3.2
  script:
    - bundle install
    - rspec --profile --format documentation
  parallel: 5
```

GitLab создаёт 5 клонов задания `test`. Каждый экземпляр выполняет примерно 20 тестов, сокращая общее время примерно на 80%.

GitLab автоматически устанавливает переменные для каждого параллельного экземпляра:
- `$CI_NODE_INDEX` — индекс текущего экземпляра (начиная с 1)
- `$CI_NODE_TOTAL` — общее количество экземпляров

Их можно использовать для кастомной логики разделения тестов в тестовых фреймворках.

---

### 11.2 `parallel:matrix` — запуск по нескольким измерениям

Когда нужно протестировать или задеплоить на разных окружениях — разных ОС, версиях runtime — `parallel:matrix` даёт чистый подход без дублирования определений заданий.

**Пример 1: различные раннеры**

```yaml
deploy-job:
  stage: deploy
  resource_group: production
  timeout: 10m
  parallel:
    matrix:
      - RUNNER_MACHINE:
          - saas-linux-small-amd64
          - saas-linux-medium-amd64
          - windows
          - macos
  tags:
    - $RUNNER_MACHINE
  image: node:20-alpine3.18
  script:
    - echo "Using runner: $RUNNER_MACHINE"
    - node -v
    - npm -v
```

Это создаёт **4** параллельных задания — по одному на каждый тип раннера.

**Пример 2: комбинация раннеров и версий Node.js**

```yaml
deploy-job:
  stage: deploy
  resource_group: production
  timeout: 10m
  parallel:
    matrix:
      RUNNER_MACHINE:
        - saas-linux-small-amd64
        - saas-linux-medium-amd64
      NODE_VERSION:
        - '20-alpine3.18'
        - '18-alpine3.18'
        - '21-alpine3.18'
  tags:
    - $RUNNER_MACHINE
  image: node:$NODE_VERSION
  script:
    - echo "Runner: $RUNNER_MACHINE, Node.js: $NODE_VERSION"
    - node -v
    - npm -v
```

Матрица создаёт **6** параллельных заданий — все комбинации 2 раннеров × 3 версий Node.js:

| Задание | RUNNER_MACHINE | NODE_VERSION |
|---|---|---|
| 1 | saas-linux-small-amd64 | 20-alpine3.18 |
| 2 | saas-linux-small-amd64 | 18-alpine3.18 |
| 3 | saas-linux-small-amd64 | 21-alpine3.18 |
| 4 | saas-linux-medium-amd64 | 20-alpine3.18 |
| 5 | saas-linux-medium-amd64 | 18-alpine3.18 |
| 6 | saas-linux-medium-amd64 | 21-alpine3.18 |

## Часть 12. Пропуск запуска пайплайна

### 12.1 Зачем пропускать пайплайны

Каждый пуш по умолчанию запускает новый пайплайн. Но для незначительных изменений — обновление документации, исправление опечаток, правки комментариев — это лишняя трата compute minutes.

**Типичные сценарии для пропуска:**
- Обновление `README.md`, `CHANGELOG.md`
- Правки конфигурации, не влияющие на сборку/тесты
- Незначительное форматирование или правки комментариев

### 12.2 Директивы пропуска

Добавьте одну из директив в **сообщение коммита**:

| Директива | Эффект |
|---|---|
| `[ci skip]` | Пайплайн не запускается |
| `[skip ci]` | Псевдоним для `[ci skip]` |

Обе директивы нечувствительны к регистру.

**Пример:**

```bash
git add README.md
git commit -m "docs: improve installation steps [ci skip]"
git push origin main
```

После пуша GitLab немедленно помечает пайплайн как **Skipped** — ни одно задание не ставится в очередь и не выполняется.

**Что показывает GitLab UI:**
- Задания: не запущены
- Compute minutes: сохранены
- Дашборд: не замусорен лишними записями

> ⚠️ **Важно:** НЕ используйте директивы пропуска для коммитов, которые изменяют скрипты сборки, тесты или production-код. Пропуск критических изменений может привести к незамеченным сбоям.

## Итоговые выводы

**1. Переменные — основа DRY-пайплайнов.** Глобальные переменные устраняют дублирование. Переменные уровня задания изолируют чувствительные данные. Предопределённые переменные дают контекст без лишней конфигурации.

**2. Маскирование переменных — обязательная практика для секретов.** Никогда не храните пароли, токены и ключи в `.gitlab-ci.yml`. Project Settings → CI/CD → Variables — правильное место для секретов.

**3. `CI_PIPELINE_SOURCE` — ключ к условному выполнению.** Понимание того, что `push`, `merge_request_event`, `schedule` — это разные источники, позволяет точно настраивать, когда и какие задания выполняются.

**4. Workflow-level rules экономят ресурсы на уровне всего пайплайна.** Если пайплайн не нужен вообще — лучше не создавать его. Это мощнее, чем отключать отдельные задания.

**5. Resource Groups — защита от параллельных деплоев.** Обязательно используйте для заданий деплоя в production. Режим `oldest_first` даёт наиболее предсказуемое поведение.

**6. Таймауты — страховка от зависших заданий.** Каждое задание с потенциально долгим выполнением должно иметь разумный таймаут.

**7. Используйте образы вместо установки инструментов.** `image: node:20-alpine3.18` — намного быстрее и надёжнее, чем устанавливать Node.js в `before_script` каждый раз.

**8. Parallel Matrix — для кросс-платформенного тестирования без дублирования.** Одно определение задания покрывает все комбинации окружений.

**9. `[ci skip]` — для незначительных изменений.** Документация не должна запускать полный пайплайн деплоя. Но используйте эту возможность осознанно — только для действительно некритичных изменений.

#### `app.js` — Backend на Express.js

```javascript
const path = require('path');
const express = require('express');
const bodyParser = require('body-parser');
const cors = require('cors');
const mongoose = require('mongoose');

const app = express();
app.use(bodyParser.json());
app.use(express.static(path.join(__dirname, '/')));
app.use(cors());

mongoose.connect(process.env.MONGO_URI, {
  user: process.env.MONGO_USERNAME,
  pass: process.env.MONGO_PASSWORD,
  useNewUrlParser: true,
  useUnifiedTopology: true
}, (err) => {
  if (err) {
    console.error("Connection error:", err);
  } else {
    console.log("Connected to MongoDB");
  }
});

app.listen(3000, () => {
  console.log("Server running on port 3000");
});
```

Приложение использует три переменных окружения для подключения к MongoDB: `MONGO_URI`, `MONGO_USERNAME`, `MONGO_PASSWORD`. Без них тесты упадут с ошибкой Mongoose.

## Часть 8. Настройка покрытия кода в пайплайне

### 8.1 Задание покрытия кода

```yaml
code_coverage:
  stage: test
  image: node:17-alpine3.14
  before_script:
    - npm install
  script:
    - npm run coverage
  artifacts:
    name: Code-Coverage-Result
    when: always
    expire_in: 3 days

        path: coverage/cobertura-coverage.xml
```

`reports.coverage_report` — говорит GitLab, что это Cobertura XML отчёт покрытия кода. GitLab отобразит покрытие в пайплайне и в merge request.

GitLab применяет regex к этой строке, извлекает число `88.88` и показывает его в UI пайплайна рядом с заданием.

## Часть 9. `allow_failure` — игнорирование сбоев задания

### 9.1 Зачем нужен `allow_failure`

По умолчанию провал задания блокирует все downstream-задания и помечает весь пайплайн как Failed. Но иногда сбой является **предупреждением**, а не критической ошибкой. Например, покрытие 88.88% при пороге 90% — это информация для разработчика, но не повод останавливать деплой.

`allow_failure: true` меняет поведение: задание остаётся как "провалившееся с предупреждением", но пайплайн продолжается.

### 9.2 Применение `allow_failure`

```yaml
code_coverage:
  stage: test
  image: node:17-alpine3.14
  before_script:
    - npm install
  script:
    - npm run coverage
  artifacts:
    name: Code-Coverage-Result
    when: always
    expire_in: 3 days
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml
  coverage: '/All files[^|]*|[^|]*\s+([\d.]+)/'
  allow_failure: true

sample-job:
  stage: test
  needs:
    - code_coverage
  image: node:17-alpine3.14
  script:
    - echo testing sample job
```

Теперь `sample-job` выполнится даже при провале `code_coverage`.

### 9.3 Управление сбоями по коду выхода

Можно ограничить `allow_failure` конкретными кодами выхода:

```yaml
test_job_1:
  script:
    - echo "Exits with code 1, so job fails"
    - exit 1
  allow_failure:
    exit_codes: [137]

test_job_2:
  script:
    - echo "Exits with code 137, allowed failure"
    - exit 137
  allow_failure:
    exit_codes:
      - 137
      - 255
```

| Задание | Разрешённые коды | Реальный код | Результат |
|---|---|---|---|
| `test_job_1` | `[137]` | 1 | Пайплайн падает (код не в списке) |
| `test_job_2` | `[137, 255]` | 137 | Провал разрешён, пайплайн продолжается с ⚠️ |

### 9.4 Как это выглядит в UI

После добавления `allow_failure: true`:
- Общий статус пайплайна: **passed** с иконкой предупреждения (⚠️)
- Задание `code_coverage` показывает "failed with warnings"
- Зависимые задания (`sample-job`) выполняются
- Артефакты (`cobertura-coverage.xml`) доступны для скачивания

```bash
$ echo testing sample job
testing sample job
```

## Часть 11. Кэширование зависимостей

### 11.1 Проблема: повторная установка при каждом запуске

При каждом запуске задания GitLab стартует в чистом контейнере. Без кэша `npm install` каждый раз устанавливает все 364+ пакета заново — это занимает около 7 секунд. При нескольких заданиях в пайплайне это складывается в значительные потери времени.

### 11.2 Cache vs Artifacts

| Характеристика | Cache | Artifacts |
|---|---|---|
| Назначение | Внешние зависимости (например, `node_modules`) | Результаты сборки или отчёты |
| Время жизни | Общий для заданий и пайплайнов, истекает по настройке | Передаётся между заданиями одного пайплайна |
| Хранение | Может храниться внешне (например, AWS S3) | Хранится в GitLab (по умолчанию) |
| Политика | `pull`, `push`, `pull-push` | Всегда загружается при успехе или неудаче |

### 11.3 Конфигурация кэша с YAML-якорями

```yaml
.default_cache: &default_cache
  key:
    files:
      - package-lock.json
    prefix: node_modules
  paths:
    - node_modules
  policy: pull-push
  when: on_success

unit_testing:
  stage: test
  image: node:17-alpine3.14
  cache: *default_cache
  before_script:
    - npm install
  script:
    - npm test

code_coverage:
  stage: test
  image: node:17-alpine3.14
  cache: *default_cache
  before_script:
    - npm install
  script:
    - npm run coverage
```

**Разбор конфигурации кэша:**

`key.files: [package-lock.json]` — ключ кэша вычисляется на основе SHA-хэша файла `package-lock.json`. Если зависимости не менялись — кэш восстановится. Если `package-lock.json` изменился — старый кэш не найдётся, произойдёт cache miss.

`prefix: node_modules` — добавляет префикс к ключу для удобной идентификации.

`policy: pull-push` — сначала восстановить существующий кэш, затем загрузить обновлённый после успешного задания.

`when: on_success` — загружать кэш только при успешном завершении задания.

Синтаксис `*default_cache` — это YAML-якоря: `&default_cache` создаёт якорь, `*default_cache` его использует. Это позволяет определить конфигурацию кэша один раз и переиспользовать в нескольких заданиях.

### 11.4 Первый запуск — cache miss

```
Restoring cache
  No cache found for key: node_modules-<sha256-of-package-lock.json>
$ npm install
added 364 packages in 7s
$ npm test ...
Saving cache for successful job
  Created cache node_modules-<sha>-non_protected
Uploading cache.zip to GitLab Runner storage...
```

### 11.5 Последующие запуски — cache hit

```
Restoring cache
  Found cache: node_modules-<sha256> ...
$ npm install
up to date, audited 365 packages in 1s
$ npm test ...
```

Время установки: **7 секунд → 1 секунда**. Это особенно заметно при многократных запусках пайплайна в течение рабочего дня.

## Часть 13. Project Status Meeting 2 — Критическая ошибка

### 13.1 Проблема: тесты запускаются против production БД

После первых успехов с пайплайном команда обнаружила, что мониторинг фиксирует замедление и нестабильность production MongoDB-кластера. **Причина:** CI/CD пайплайн напрямую подключается к production БД через переменные окружения:

```yaml
variables:
  MONGO_URI: 'mongodb+srv://supercluster.d3jj.mongodb.net/superData'
  MONGO_USERNAME: superuser
  MONGO_PASSWORD: $M_DB_PASSWORD
```

Оба задания — `unit_testing` и `code_coverage` — стучатся в production!

> ⚠️ **Никогда не запускайте CI-тесты против production базы данных.** Это может привести к деградации производительности, повреждению данных и рискам безопасности.

### 13.2 Последствия тестов против production

- Дополнительная нагрузка и задержки для реальных пользователей
- Случайное изменение или удаление критических данных
- Усложнение отладки из-за общего состояния

### 13.3 Решение: GitLab CI Services

GitLab CI/CD поддерживает **services** — вспомогательные контейнеры, запускаемые параллельно с вашим заданием. Это позволяет:

- Запустить контейнер с базой данных (например, `mongo:4.4`)
- Заполнить её тестовыми данными в `before_script`
- Направить приложение на `localhost` или алиас сервиса

**Рефакторинг пайплайна с MongoDB сервисом:**

```yaml
variables:
  MONGO_INITDB_DATABASE: testdb
  MONGO_INITDB_ROOT_USERNAME: testuser
  MONGO_INITDB_ROOT_PASSWORD: testpass

.default_test_template: &test_template
  stage: test
  image: node:17-alpine3.14
  services:
    - name: mongo:4.4
      alias: mongo
  before_script:
    - npm install
    - |
      until mongo --host mongo -u $MONGO_INITDB_ROOT_USERNAME \
        -p $MONGO_INITDB_ROOT_PASSWORD \
        --eval "db.adminCommand('ping')" &>/dev/null; do
        echo "Waiting for MongoDB..."
        sleep 2
      done
    - npm run seed-test-data
  variables:
    MONGO_URI: "mongodb://$MONGO_INITDB_ROOT_USERNAME:$MONGO_INITDB_ROOT_PASSWORD@mongo:27017/$MONGO_INITDB_DATABASE"

unit_testing:
  <<: *test_template
  script:
    - npm test
  artifacts:
    when: always
    expire_in: 3 days
    name: moca-test-result
    paths:
      - test-results.xml
    reports:
      junit: test-results.xml

code_coverage:
  <<: *test_template
  script:
    - npm run coverage
  artifacts:
    when: always
    expire_in: 3 days
    name: coverage-report
    paths:
      - coverage/
```

**Сравнение до и после рефакторинга:**

| Задание | Исходная БД | После рефакторинга |
|---|---|---|
| `unit_testing` | Production MongoDB кластер | Локальный `mongo:4.4` сервис |
| `code_coverage` | Production MongoDB кластер | Локальный `mongo:4.4` сервис |

Конструкция `<<: *test_template` — это YAML merge key, позволяющий наследовать всю конфигурацию из шаблона `&test_template`. Это DRY-подход: общая конфигурация определена один раз, а задания только добавляют специфические шаги.

## Итоговые выводы

**1. Реальный проект требует реальной инфраструктуры.** Solar System показывает, что CI/CD — это не просто запуск `echo` команд. Нужна БД, переменные окружения, корректные пороги покрытия.

**2. Структура `package.json` определяет CI конфигурацию.** Скрипты `test` и `coverage`, зависимости и пороги nyc — всё это напрямую влияет на то, как строятся задания пайплайна.

**3. Маскированные переменные — обязательная практика для учётных данных.** `$M_DB_PASSWORD` в YAML ссылается на защищённую переменную в настройках проекта, а не на реальный пароль.

**4. `reports:junit` превращает XML в живую аналитику.** Это не просто сохранение файла — GitLab читает его и показывает результаты тестов в pipeline UI и merge request.

**5. Regex для покрытия — мощный инструмент.** Правильное регулярное выражение в директиве `coverage:` позволяет GitLab автоматически извлекать процент покрытия из логов любого инструмента.

**6. `allow_failure: true` — для некритичных проверок.** Покрытие кода — важный показатель, но небольшое отклонение не должно останавливать деплой. `allow_failure` превращает ошибки в предупреждения.

**7. Кэширование зависимостей — простая оптимизация с большим эффектом.** 7 секунд → 1 секунда за один приём. Ключ на основе `package-lock.json` обеспечивает автоматическую инвалидацию при изменении зависимостей.

**8. Production БД никогда не должна использоваться в тестах.** GitLab Services позволяет запустить изолированный MongoDB контейнер прямо в задании — быстро, безопасно и без рисков для реальных данных.

## Часть 1. Сервисы (Services) — концепция и архитектура

### 1.1 Что такое Services в GitLab CI/CD

**Services** — это вспомогательные Docker-контейнеры, которые запускаются **параллельно** с основным контейнером задания и доступны ему по сети. Они предназначены для сетевых зависимостей: баз данных, очередей сообщений, кэш-серверов — всего, к чему основной контейнер задания должен подключаться по сети.

Ключевое понимание: оба контейнера — основной и сервисный — работают в одной Docker-сети (shared bridge network), поэтому могут общаться между собой по имени хоста.

---

### 1.2 Чем Services НЕ являются — критически важное замечание

Распространённое заблуждение: разработчики пытаются добавить в `services:` языковые рантаймы или CLI-инструменты. Это **не работает**:

```yaml
job:
  services:
    - php:7
    - node:latest
    - golang:1.10
  image: alpine:3.7
  script:
    - php -v       # ❌ Ошибка: нет PHP-бинарника
    - node -v      # ❌ Ошибка: нет Node.js
    - go version   # ❌ Ошибка: нет Go
```

Почему это не работает: образы в `services:` запускаются как **отдельные контейнеры** — их бинарники недоступны в основном контейнере задания. Они только слушают сетевые порты.

**Правило:** языковые рантаймы и инструменты сборки — в `image:`, сетевые зависимости (БД, API) — в `services:`.

---

### 1.3 Подключение к сервисам — алиасы и хостнеймы

По умолчанию сервис доступен по двум автогенерируемым хостнеймам:

- Замена `/` на `__` (двойное подчёркивание): `tutum__wordpress`
- Замена `/` на `-` (одинарный дефис): `tutum-wordpress`

Часть с тегом (после `:`) отбрасывается. Поскольку подчёркивания не являются валидными по RFC 1123 (стандарт DNS-имён), **рекомендуется всегда задавать явный алиас**:

```yaml
job:
  image: alpine:3.7
  services:
    - name: tutum/wordpress:latest
      alias: wordpress
  script:
    - curl http://wordpress
```

---

### 1.4 Полные параметры конфигурации Services

| Параметр | Обязательный | Описание |
|---|---|---|
| `name` | Да | Полное имя образа включая registry и тег |
| `alias` | Нет | Кастомное имя хоста для контейнера сервиса |
| `entrypoint` | Нет | Переопределить Docker entrypoint |
| `command` | Нет | Переопределить Docker command |
| `variables` | Нет | Переменные окружения для контейнера сервиса |
| `pull_policy` | Нет | Политика загрузки образа: `always`, `if-not-present` |

**Пример с расширенной конфигурацией:**

```yaml
default:
  image: ruby:2.6
  entrypoint: ["bash"]

services:
  - name: my-postgres:11.7
    alias: db-postgres
    entrypoint: ["/usr/local/bin/docker-entrypoint.sh"]
    command: ["postgres"]
    variables:
      POSTGRES_DB: test
      POSTGRES_USER: user
      POSTGRES_PASSWORD: secret
```

**Политики загрузки образов:**

```yaml
job1:
  script: echo "Использую if-not-present"
  services:
    - name: postgres:11.6
      pull_policy: if-not-present  # загружать только если нет локально

job2:
  script: echo "Всегда загружать"
  services:
    - name: postgres:11.6
      pull_policy: [always, if-not-present]  # сначала попробовать always, fallback
```

---

### 1.5 Варианты предоставления инструментов заданию

GitLab предлагает несколько подходов для обеспечения заданий нужными инструментами:

**Подход 1: Теги раннеров**

```yaml
unit-testing:
  tags:
    - ubuntu-latest
  script:
    - echo "Running unit tests on Ubuntu"
```

Теги направляют задание на конкретный раннер с предустановленными инструментами. Раннер `ubuntu-latest` предоставляет полное Ubuntu-окружение.

**Подход 2: Установка в `before_script` (медленно, но работает)**

```yaml
node-test:
  tags:
    - sast-linux-small-amd64
  before_script:
    - apt-get update && apt-get install -y curl gnupg
    - curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
    - apt-get install -y nodejs
  script:
    - npm install
    - npm test
```

Минус: каждый запуск задания повторяет установку (~2-5 минут дополнительного времени).

**Подход 3: Docker-образ (рекомендуется)**

```yaml
node-test:
  tags:
    - linux
  image: node:20-alpine3.17
  script:
    - npm install
    - npm test
```

**Сравнительная таблица подходов:**

| Конфигурация | Шаги установки | Продолжительность пайплайна | Операционная нагрузка |
|---|---|---|---|
| Установка в `before_script` | Обновление ОС, добавление репозиториев, установка | Высокая | Высокая |
| Docker-образ | Загрузка готового контейнера | Низкая | Низкая |

## Часть 3. Docker Build — сборка образов в пайплайне

### 3.1 Docker-in-Docker (DinD)

Для сборки Docker-образов внутри GitLab CI используется технология **Docker-in-Docker (DinD)** — запуск Docker daemon внутри Docker-контейнера. Это стандартный подход для CI/CD пайплайнов.

**Как это работает:** основной контейнер задания использует образ `docker:24.0.5` (Docker CLI), а сервис `docker:24.0.5-dind` предоставляет Docker daemon, к которому подключается CLI.

### 3.2 Настройка `docker_build` задания

Добавляем стадию `containerization`:

```yaml
stages:
  - test
  - containerization

variables:
  DOCKER_USERNAME: siddharth67
  IMAGE_VERSION: $CI_PIPELINE_ID
  MONGO_URI: mongodb+srv://supercluster.d83jj.mongodb.net/superData
  MONGO_USERNAME: superuser
  MONGO_PASSWORD: $M_DB_PASSWORD

docker_build:
  stage: containerization
  image: docker:24.0.5
  services:
    - docker:24.0.5-dind
  dependencies: []
  rules:
    - if: $CI_COMMIT_BRANCH == "main" || $CI_COMMIT_BRANCH =~ /^feature/
      when: always
    - if: $CI_MERGE_REQUEST_SOURCE_BRANCH_NAME =~ /^feature/ && $CI_PIPELINE_SOURCE == "merge_request_event"
      when: always
  script:
    - docker build -t $DOCKER_USERNAME/solar-system:$IMAGE_VERSION .
    - docker images $DOCKER_USERNAME/solar-system:$IMAGE_VERSION
```

**Разбор ключевых параметров:**

`image: docker:24.0.5` — официальный Docker-образ, содержащий Docker CLI. Именно в этом контейнере будут выполняться docker-команды.

`services: docker:24.0.5-dind` — Docker daemon, запущенный как сервис. CLI в основном контейнере подключается к нему.

`dependencies: []` — пустой список зависимостей означает, что задание не скачивает артефакты от предыдущих заданий. Это оптимизация: нет смысла скачивать `test-results.xml` для сборки Docker-образа.

`IMAGE_VERSION: $CI_PIPELINE_ID` — тег образа основан на ID пайплайна. Каждый запуск создаёт уникальный тег, что обеспечивает трассируемость.

**Логи сборки Docker-образа:**

## Часть 4. Docker Test — проверка образа перед пушем

### 4.1 Зачем тестировать Docker-образ

Собранный образ может содержать проблемы, которые не выявляются при запуске юнит-тестов:

- Некорректные переменные окружения
- Проблемы с сетевой конфигурацией внутри контейнера
- Неработающие эндпоинты health check
- Конфликты портов или зависимостей

Задание `docker_test` запускает контейнер и проверяет, что приложение реально отвечает на запросы — **до** того, как образ попадёт в registry.

### 4.2 Сохранение образа как артефакта

Чтобы передать образ между заданиями одного пайплайна — сохраняем его как tar-архив:

```yaml
docker_build:
  stage: containerization
  image: docker:24.0.5
  services:
    - docker:24.0.5-dind
  script:
    - docker build -t $DOCKER_USERNAME/solar-system:$IMAGE_VERSION .
    - docker images $DOCKER_USERNAME/solar-system:$IMAGE_VERSION
    - mkdir -p image
    - docker save $DOCKER_USERNAME/solar-system:$IMAGE_VERSION \
        -o image/solar-system-image-$IMAGE_VERSION.tar
  artifacts:
    paths:
      - image
    when: on_success
    expire_in: 3 days
```

`docker save` экспортирует образ в tar-архив. `artifacts.paths: [image]` загружает всю директорию `image/` на GitLab. Последующие задания автоматически скачивают этот артефакт.

### 4.3 Задание `docker_test`

```yaml
docker_test:
  stage: containerization
  image: docker:24.0.5
  needs:
    - docker_build
  services:
    - docker:24.0.5-dind
  script:
    - docker load -i image/solar-system-image-$IMAGE_VERSION.tar
    - docker run --name solar-system-app -d -p 3000:3000 \
        $DOCKER_USERNAME/solar-system:$IMAGE_VERSION
    - export IP=$(docker inspect -f \
        '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' solar-system-app)
    - echo "App IP: $IP"
    - docker run --rm alpine \
        wget -qO- http://$IP:3000/live | grep -q '"live"'
```

**Пошаговый разбор скрипта:**

**Шаг 1 — `docker load`:** импортирует tar-архив обратно в Docker. Без этого шага образ недоступен — он не был загружен из registry, а был создан в предыдущем задании.

**Шаг 2 — `docker run`:** запускает контейнер в detached режиме (`-d`) на порту 3000. `--name solar-system-app` даёт контейнеру имя для последующего обращения.

**Шаг 3 — `docker inspect`:** использует Go-шаблон для извлечения IP-адреса запущенного контейнера. Так как DinD работает в изолированной Docker-сети, нельзя обратиться к контейнеру через `localhost` — нужен реальный IP.

**Шаг 4 — `wget` из Alpine:** запускает одноразовый (`--rm`) Alpine-контейнер, который делает GET-запрос к `/live` эндпоинту и проверяет наличие строки `"live"` в ответе. Если проверка не прошла — `grep -q` возвращает ненулевой код выхода, задание падает.

### 4.4 Health-check эндпоинты в `app.js`

```javascript
app.get('/live', (req, res) => {
  res.json({ status: "live" });
});

app.get('/ready', (req, res) => {
  res.json({ status: "ready" });
});

app.get('/os', (req, res) => {
  res.json({ os: OS.hostname(), env: process.env.NODE_ENV });
});
```

`/live` — liveness probe: приложение запущено и отвечает на запросы.
`/ready` — readiness probe: приложение готово принимать трафик (можно добавить проверку подключения к БД).

Эти же эндпоинты используются в Kubernetes для health checks подов.

## Часть 5. Docker Push — публикация образа в Docker Hub

### 5.1 Конфигурация `docker_push`

```yaml
docker_push:
  stage: containerization
  needs:
    - docker_build
    - docker_test
  image: docker:24.0.5
  services:
    - docker:24.0.5-dind
  script:
    - docker load -i image/solar-system-image-$IMAGE_VERSION.tar
    - docker login --username=$DOCKER_USERNAME --password=$DOCKER_PASSWORD
    - docker push $DOCKER_USERNAME/solar-system:$IMAGE_VERSION
```

`needs: [docker_build, docker_test]` — задание запустится только после успешного завершения обоих предшественников. Если `docker_test` упал — `docker_push` не выполнится, "сломанный" образ не попадёт в registry.

### 5.2 Настройка переменных в GitLab

**Settings → CI/CD → Variables:**

| Имя переменной | Значение | Masked | Описание |
|---|---|---|---|
| `DOCKER_USERNAME` | `siddharth67` | Нет | Docker Hub username |
| `DOCKER_PASSWORD` | `<ваш_пароль>` | Да | Docker Hub пароль или access token |
| `M_DB_PASSWORD` | `<mongodb_pass>` | Да | Пароль MongoDB |

> ⚠️ **Никогда не хардкодьте учётные данные в `.gitlab-ci.yml`.** Всегда используйте маскированные переменные CI/CD.

## Часть 6. GitLab Container Registry — встроенный registry

### 6.1 Что такое GitLab Container Registry

**GitLab Container Registry** — это встроенный приватный Docker registry, доступный для каждого GitLab-проекта. Преимущество перед Docker Hub: образы хранятся рядом с кодом, в той же платформе, без необходимости внешнего сервиса.

**Методы аутентификации:**
- Personal Access Tokens
- Deploy Tokens
- Project Access Tokens
- Group Access Tokens

### 6.2 Предопределённые переменные для registry

GitLab автоматически предоставляет переменные для работы с Container Registry:

| Переменная | Описание | Пример |
|---|---|---|
| `CI_REGISTRY` | Хостнейм registry | `registry.gitlab.com` |
| `CI_REGISTRY_USER` | Username для Docker login | автоматически |
| `CI_REGISTRY_PASSWORD` | Пароль или токен аутентификации | автоматически |
| `CI_REGISTRY_IMAGE` | Полный путь к образу проекта | `registry.gitlab.com/group/project` |

Ни одну из этих переменных не нужно настраивать вручную — они доступны в каждом задании автоматически.

**Аутентификация в CI-задании:**

```bash
# Вариант 1: с CI_JOB_TOKEN
docker login $CI_REGISTRY -u $CI_REGISTRY_USER --password-stdin <<< "$CI_JOB_TOKEN"

# Вариант 2: с CI_REGISTRY_PASSWORD
docker login $CI_REGISTRY -u $CI_REGISTRY_USER --password-stdin <<< "$CI_REGISTRY_PASSWORD"
```

**Локальная авторизация с токеном:**

```bash
TOKEN=<your_token>
docker login registry.example.com -u <username> --password-stdin <<< "$TOKEN"
```

### 6.3 Задание публикации в GitLab Container Registry

```yaml
publish_gitlab_container_registry:
  stage: containerization
  needs:
    - docker_build
    - docker_test
  image: docker:24.0.5
  services:
    - docker:24.0.5-dind
  script:
    - docker load -i image/solar-system-image-$CI_PIPELINE_ID.tar
    - echo "Registry: $CI_REGISTRY | User: $CI_REGISTRY_USER | Repo: $CI_REGISTRY_IMAGE"
    - docker login $CI_REGISTRY -u "$CI_REGISTRY_USER" -p "$CI_REGISTRY_PASSWORD"
    - docker tag $DOCKER_USERNAME/solar-system:$IMAGE_VERSION $CI_REGISTRY_IMAGE/ss-image:$IMAGE_VERSION
    - docker images
    - docker push $CI_REGISTRY_IMAGE/ss-image:$IMAGE_VERSION
```

**Разбор ключевых шагов:**

`docker load` — восстанавливает образ из tar-архива артефакта.

`docker tag` — создаёт новый тег для образа с путём GitLab registry. Образ `siddharth67/solar-system:123` переименовывается в `registry.gitlab.com/demos-group/solar-system/ss-image:123`.

`docker push` — публикует образ в GitLab Container Registry.

### 6.4 Работа с двумя registry одновременно

Интересный паттерн: один образ может публиковаться **и в Docker Hub, и в GitLab Registry** в рамках одного пайплайна. Это обеспечивает избыточность и гибкость:

```yaml
# docker_push публикует в Docker Hub
docker_push:
  script:
    - docker push $DOCKER_USERNAME/solar-system:$IMAGE_VERSION

# publish_gitlab_container_registry публикует в GitLab Registry
publish_gitlab_container_registry:
  script:
    - docker push $CI_REGISTRY_IMAGE/ss-image:$IMAGE_VERSION
```

Оба задания зависят от `docker_build` и `docker_test`, и могут выполняться **параллельно** друг с другом.

### 6.5 Устранение проблем с аутентификацией

Если возникает ошибка синтаксиса с `--password-stdin`, используйте флаг `-p`:

```yaml
# Проблемный вариант:
- docker login $CI_REGISTRY -u "$CI_REGISTRY_USER" --password-stdin <<< "$CI_REGISTRY_PASSWORD"

# Рабочий альтернатива:
- docker login $CI_REGISTRY -u "$CI_REGISTRY_USER" -p "$CI_REGISTRY_PASSWORD"
```

> ⚠️ Передача пароля через `-p` менее безопасна (пароль может попасть в историю команд). Используйте `--password-stdin` где возможно.

```bash
WARNING! Using --password via the CLI is insecure.
```

## Итоговые выводы

**1. Services — для сетевых зависимостей, `image:` — для инструментов.** Это фундаментальное разделение. MongoDB в `services:`, Node.js в `image:`.

**2. `alias` для сервисов — обязательная практика.** Автогенерируемые имена с `__` ненадёжны. Явный `alias: mongo` всегда лучше.

**3. Docker-in-Docker — стандарт для сборки образов в CI.** Пара `image: docker:24.0.5` + `services: docker:24.0.5-dind` — проверенный подход.

**4. Сохранение образа как артефакт — мост между заданиями.** `docker save` → артефакт → `docker load` в следующем задании позволяет строить цепочку build → test → push без повторной сборки.

**5. Тестирование образа перед пушем — обязательный шаг.** Запуск контейнера и проверка `/live` эндпоинта занимают секунды, но спасают от публикации нерабочих образов.

**6. `needs` в Docker-заданиях критичен для порядка.** `docker_push` должен зависеть от `docker_test`. Без этого можно запушить непроверенный образ.

**7. GitLab Container Registry — удобная альтернатива Docker Hub.** Предопределённые переменные `CI_REGISTRY_*` делают аутентификацию тривиальной, образы хранятся рядом с кодом.

**8. Изоляция тестовой БД устраняет производственные проблемы.** Разница в нагрузке на production MongoDB: с 40 одновременными CI-подключениями vs нулем после рефакторинга — огромная.

# Оптимизация GitLab CI/CD — DRY-пайплайны, безопасность, мониторинг и интеграции

## Часть 1. Оптимизация конфигурационных файлов CI/CD

### 1.1 Проблема копипасты в пайплайнах

При быстром старте команды часто копируют блоки конфигурации из одного задания в другое. Это работает на начальном этапе, но создаёт серьёзные проблемы при масштабировании:

- Изменение одной настройки требует правок в десяти местах
- Легко пропустить обновление в одном из заданий
- Трудно отследить, какие задания используют "актуальную" конфигурацию

GitLab CI/CD предоставляет три механизма для создания DRY-пайплайнов:

| Механизм | Назначение | Область | Кросс-файловая поддержка | Синтаксис |
|---|---|---|---|---|
| `extends` | Наследование и переопределение определений заданий | Несколько файлов | Да | `extends: .base_job` |
| YAML Anchors | Повторное использование блоков внутри одного файла | Один файл | Нет | `&anchor` / `<<: *anchor` |
| Reference Tags | Извлечение фрагментов из YAML-файлов | Несколько файлов | Да | `!reference [job, key]` |

---

### 1.2 Скрытые задания (Hidden Jobs)

**Скрытые задания** — это задания с именем, начинающимся с точки (`.`). Они **никогда не выполняются самостоятельно** и служат только как шаблоны для повторного использования.

```yaml
.hidden_common:
  cache:
    policy: pull-push
    key:
      files: [package.json]
      paths: [node_modules]
  before_script:
    - npm install
```

Сценарии использования скрытых заданий:
- Общие правила кэша
- Общие шаги `before_script`
- Отключение заданий без удаления (добавляем точку перед именем)

## Часть 2. Ключевое слово `extends` — наследование конфигурации

### 2.1 Базовый синтаксис `extends`

`extends` позволяет заданию **наследовать** конфигурацию из скрытого или другого задания. GitLab **сливает** (merge) родительское и дочернее определения, причём значения дочернего задания переопределяют дублирующиеся ключи из родительского.

```yaml
.tests:
  rules:
    - if: $CI_PIPELINE_SOURCE == "push"

.rspec:
  extends: .tests
  script:
    - rake rspec

rspec 1:
  extends: .rspec
  variables:
    RSPEC_SUITE: '1'

rspec 2:
  extends: .rspec
  variables:
    RSPEC_SUITE: '2'
```

GitLab поддерживает до 11 уровней наследования, но рекомендуется не превышать 2-3 уровней — глубокие деревья наследования тяжело поддерживать.

### 2.2 Наследование от нескольких шаблонов

```yaml
.only-important:
  variables:
    URL: "http://my-url.internal"
    IMPORTANT_VAR: "details"
  rules:
    - if: $CI_COMMIT_BRANCH == $CI_DEFAULT_BRANCH
    - if: $CI_COMMIT_BRANCH == "stable"
  tags:
    - production
  script:
    - echo "Hello world!"

.in-docker:
  image: alpine
  variables:
    URL: "http://docker-url.internal"
  tags:
    - docker

rspec:
  extends:
    - .only-important
    - .in-docker
  variables:
    GITLAB: "is-awesome"
  script:
    - rake rspec
```

При конфликтах между несколькими родителями применяется приоритет: последний родитель в списке имеет больший приоритет, а дочернее задание переопределяет всё.

### 2.3 Практический рефакторинг: Node.js тестовые задания

**Исходная проблема** — два задания дублируют огромный блок конфигурации:

```yaml
unit_testing:
  stage: test
  image: node:17-alpine3.14
  services:
    - name: siddharth67/mongo-db:non-prod
      alias: mongo
      pull_policy: always
  variables:
    MONGO_URI: 'mongodb://mongo:27017/superData'
    MONGO_USERNAME: non-prod-user
    MONGO_PASSWORD: non-prod-password
  cache:
    key:
      files:
        - package-lock.json
      prefix: node_modules
    policy: pull-push
    when: on_success
    paths:
      - node_modules
  before_script:
    - npm install
  script:
    - npm test

code_coverage:
  # Точно такой же блок конфигурации...
  script:
    - npm run coverage
```

**Шаг 1: Создаём скрытый шаблон**

```yaml
.prepare_nodejs_environment:
  image: node:17-alpine3.14
  services:
    - name: siddharth67/mongo-db:non-prod
      alias: mongo
      pull_policy: always
  variables:
    MONGO_URI: 'mongodb://mongo:27017/superData'
    MONGO_USERNAME: non-prod-user
    MONGO_PASSWORD: non-prod-password
  cache:
    policy: pull-push
    when: on_success
    paths:
      - node_modules
    key:
      files:
        - package-lock.json
      prefix: node_modules
  before_script:
    - npm install
```

**Шаг 2: Рефакторинг заданий через `extends`**

```yaml
unit_testing:
  stage: test
  extends: .prepare_nodejs_environment
  script:
    - npm test
  artifacts:
    when: always
    expire_in: 3 days
    name: Moca-Test-Results
    paths:
      - test-results.xml
    reports:
      junit: test-results.xml

code_coverage:
  stage: test
  extends: .prepare_nodejs_environment
  script:
    - npm run coverage
  artifacts:
    name: Code-Coverage-Report
    when: always
    expire_in: 3 days
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml
  coverage: /All files[^|]*\|[^|]*\s+(\d+\.\d+)/
  allow_failure: true
```

Теперь весь общий код находится в одном месте. Нужно поменять образ Node.js? Меняем одну строку в `.prepare_nodejs_environment` — оба задания автоматически получат обновление.

## Часть 3. YAML Anchors — повторное использование внутри файла

### 3.1 Синтаксис YAML Anchors

```yaml
# & создаёт якорь
.default_scripts: &default_scripts
  - ./default-script1.sh
  - ./default-script2.sh

# * использует якорь
job1:
  script:
    - *default_scripts
    - ./job-script.sh
```

| Синтаксис | Описание |
|---|---|
| `&name` | Определяет якорь с именем |
| `*name` | Вставляет содержимое именованного якоря |
| `<<: *name` | Сливает (merge) содержимое якоря в текущий объект |

### 3.2 Слияние целых конфигураций заданий

```yaml
.job_template: &job_configuration
  image: ruby:2.6
  services:
    - postgres
    - redis

test1:
  <<: *job_configuration
  script:
    - echo "Running test1"

test2:
  <<: *job_configuration
  script:
    - echo "Running test2"
```

`test1` и `test2` наследуют `image: ruby:2.6` и оба сервиса. Каждое задание добавляет только свой уникальный `script`.

### 3.3 Комбинирование нескольких якорей

Можно разбить конфигурацию на несколько якорей и выборочно их применять:

```yaml
.job_template: &job_configuration
  script:
    - echo "Test project"
  tags:
    - dev

.postgres_services:
  services: &postgres_configuration
    - postgres
    - ruby

.mysql_services:
  services: &mysql_configuration
    - mysql
    - ruby

test_postgres:
  <<: *job_configuration
  services: *postgres_configuration
  tags:
    - postgres

test_mysql:
  <<: *job_configuration
  services: *mysql_configuration
```

`test_postgres` наследует базовый `script` и `tags`, затем переопределяет `services`. `test_mysql` делает то же самое, но с MySQL.

### 3.4 Реальный пример: шаблон для деплойментов в Kubernetes

Оба задания `k8s_dev_deploy` и `k8s_stage_deploy` используют:
- `alpine:3.7` как образ
- Пустой список `dependencies`
- Идентичные шаги `before_script` для установки `kubectl` и `gettext`

**Определяем шаблон с якорем:**

```yaml
.prepare_deployment_environment: &kubernetes_deploy_job
  image:
    name: alpine:3.7
  dependencies: []
  before_script:
    - wget "https://storage.googleapis.com/kubernetes-release/release/$(wget -q -O - \
       https://storage.googleapis.com/kubernetes-release/release/stable.txt)/bin/linux/amd64/kubectl"
    - chmod +x ./kubectl
    - mv ./kubectl /usr/bin/kubectl
    - apk add --no-cache gettext
    - envsubst -V
```

**Задание деплоя в dev:**

```yaml
k8s_dev_deploy:
  <<: *kubernetes_deploy_job
  stage: dev-deploy
  needs:
    - docker_push
  script:
    - export KUBECONFIG=$DEV_KUBE_CONFIG
    - kubectl version -o yaml
    - kubectl config get-contexts
    - kubectl get nodes
    - export INGRESS_IP=$(kubectl -n ingress-nginx \
        get services ingress-nginx-controller \
        -o jsonpath="{.status.loadBalancer.ingress[0].ip}")
    - echo $INGRESS_IP
    - kubectl -n $NAMESPACE create secret generic mongo-db-creds \
        --from-literal=MONGO_URI=$MONGO_URI \
        --from-literal=MONGO_USERNAME=$MONGO_USERNAME \
        --from-literal=MONGO_PASSWORD=$MONGO_PASSWORD \
        --save-config --dry-run=client -o yaml | kubectl apply -f -
```

**Задание деплоя в staging:**

```yaml
k8s_stage_deploy:
  <<: *kubernetes_deploy_job
  stage: stage-deploy
  when: manual
  script:
    - temp_kube_config_file=$(printenv KUBECONFIG)
    - cat $temp_kube_config_file
    - kubectl config get-contexts
    - kubectl config use-context demos-group/solar-system:kk-gitlab-agent
    - kubectl get po -A
    - export INGRESS_IP=$(kubectl -n ingress-nginx \
        get services ingress-nginx-controller \
        -o jsonpath="{.status.loadBalancer.ingress[0].ip}")
    - echo $INGRESS_IP
    - kubectl -n $NAMESPACE create secret generic mongo-db-creds \
        --from-literal=MONGO_URI=$MONGO_URI \
        --from-literal=MONGO_USERNAME=$MONGO_USERNAME \
        --from-literal=MONGO_PASSWORD=$MONGO_PASSWORD \
        --save-config --dry-run=client -o yaml | kubectl apply -f -
```

Оба задания получают `image`, `dependencies` и `before_script` из якоря, добавляя только свою специфическую логику деплоя.

> ⚠️ **Важное ограничение:** YAML-якоря работают **только внутри одного файла**. Для кросс-файлового повторного использования нужны `extends` или `!reference`.

## Часть 4. Reference Tags (`!reference`) — точечный импорт между файлами

### 4.1 Синтаксис `!reference`

`!reference` — это кастомный тег GitLab CI/CD, который импортирует **конкретные секции** из других заданий или включённых файлов:

```yaml
!reference [имя_задания, имя_секции]
```

Отличие от YAML-якорей: `!reference` работает **между файлами** через `include`, позволяя точечно импортировать только нужные части конфигурации.

### 4.2 Базовый пример с `!reference`

```yaml
# setup.yml
.setup:
  script:
    - echo "creating environment"

# .gitlab-ci.yml
include:
  - local: setup.yml

.teardown:
  after_script:
    - echo "deleting environment"

test:
  script:
    - !reference [.setup, script]
```

### 4.3 Практический пример: интеграционные тесты

Два задания — `k8s_dev_integration_testing` и `k8s_stage_integration_testing` — имеют идентичные `before_script` и `script`. Используем `!reference` для исключения дублирования.

**Сначала определяем скрытый шаблон для Kubernetes-окружения:**

```yaml
.prepare_deployment_environment: &kubernetes_deploy_job
  image:
    name: alpine:3.7
  dependencies: []
  before_script:
    - wget https://storage.googleapis.com/kubernetes-release/release/...
    - chmod +x ./kubectl && mv ./kubectl /usr/bin/kubectl
    - apk add --no-cache gettext
    - envsubst --version
```

**Задание dev-интеграционных тестов (эталон):**

```yaml
k8s_dev_integration_testing:
  stage: dev-deploy
  image: !reference [.prepare_deployment_environment, image]
  needs:
    - k8s_dev_deploy
  before_script:
    - apk --no-cache add curl jq
  script:
    - echo "$INGRESS_URL"
    - curl -s -k "https://$INGRESS_URL/live"  | jq -r .status | grep -i live
    - curl -s -k "https://$INGRESS_URL/ready" | jq -r .status | grep -i ready
```

**Задание stage-интеграционных тестов (ссылается на dev):**

```yaml
k8s_stage_integration_testing:
  stage: stage-deploy
  image: !reference [.prepare_deployment_environment, image]
  needs:
    - k8s_stage_deploy
  before_script: !reference [k8s_dev_integration_testing, before_script]
  script:        !reference [k8s_dev_integration_testing, script]
```

Теперь `k8s_stage_integration_testing` автоматически синхронизирован с `k8s_dev_integration_testing`. Если изменить `before_script` или `script` в dev-задании — stage-задание получит обновление автоматически.

**Полный пример с артефактами и окружением:**

```yaml

  stage: stage-deploy

        - app_ingress_url.env

    name: staging
    url: "https://$INGRESS_URL"
```

## Часть 5. Шаблоны и типы Include

### 5.1 Типы GitLab CI/CD Templates

GitLab предоставляет два типа шаблонов:

- **Pipeline Templates** — полные end-to-end CI/CD воркфлоу для распространённых типов проектов (Node.js, Ruby on Rails и т.д.)
- **Job Templates** — отдельные задания для конкретных задач: сканирование безопасности, линтинг, Docker-сборки

Включая шаблон, вы наследуете предопределённые стадии и задания, переопределяя только то, что уникально для вашего проекта.

**Сценарий с двумя командами:**

- **Команда A:** registry Docker Hub, деплой на AWS EKS
- **Команда B:** registry Google Container Registry, деплой на GKE

Обе команды используют один Node.js шаблон, но инжектируют разные переменные, учётные данные и переопределения `script`.

### 5.2 Четыре типа `include`

| Тип include | Описание | Пример YAML |
|---|---|---|
| `local` | Файлы в том же репозитории и ветке | `- local: 'jobs/.after-script.yml'` |
| `remote` | YAML из внешнего URL | `- remote: 'https://example.com/ci/.before-script.yml'` |
| `project` | Файлы из другого проекта на том же инстансе | `- project: 'my-group/avengers-project'` + `ref: main` + `file: '/jobs/.gitlab-ci.yml'` |
| `template` | Встроенные CI-шаблоны GitLab | `- template: 'Code-Quality.gitlab-ci.yml'` |

**Условные include через `rules`:**

```yaml
include:
  - template: 'Code-Quality.gitlab-ci.yml'
    rules:
      - if: '$CI_COMMIT_BRANCH =~ /^feature\//'
        when: always
```

Шаблон Code-Quality включается только для feature-веток.

> ⚠️ При использовании `local` include по умолчанию используется ветка `HEAD`. Укажите `ref`, если нужна другая ветка или тег.

## Часть 6. Code Quality — анализ качества кода

### 6.1 Что такое Code Quality в GitLab

GitLab's Code Quality использует движок **Code Climate** (open-source) плюс дополнительные сканеры. Он анализирует:
- Сложность кода
- Дублирование
- Стилевые проблемы
- Риски поддерживаемости

**Поддерживаемые языки:** Ruby, Python, PHP, JavaScript и многие другие через плагины Code Climate.

**Возможности по тарифам:**

| Тарифный план | Ключевые возможности |
|---|---|
| Free | Базовые проверки поддерживаемости и стиля |
| Premium | Конфигурация кастомных сканеров, артефакты отчётов |
| Ultimate | Quality dashboard, расширенная аналитика, инсайты в MR |

### 6.2 Добавление Code Quality через компонент

Используем официальный компонент из CI/CD Catalog:

```yaml
include:
  - component: gitlab.com/gitlab-components/code-quality/code-quality@1.0

stages:
  - test

.prepare_nodejs_environment:
  image: node:14
  before_script:
    - npm install

unit_testing:
  stage: test
  extends: .prepare_nodejs_environment
  script:
    - npm test
  artifacts:
    when: always
    expire_in: 3 days
    reports:
      coverage_report:
        coverage_format: cobertura
        path: coverage/cobertura-coverage.xml
```

После коммита пайплайн автоматически добавит задание `code_quality`.

### 6.3 Итоговая конфигурация задания code_quality

Просмотрев **Full configuration**, увидим как компонент разворачивается:

```yaml
code_quality:
  stage: test
  image: docker:20.10.12
  allow_failure: true
  services:
    - name: docker:20.10.12-dind
      command: ["--tls=false", "--host=tcp://0.0.0.0:2375"]
  variables:
    DOCKER_DRIVER: overlay2
    DOCKER_TLS_CERTDIR: ""
  script:
    - export SOURCE_CODE=$PWD
    - docker pull -q $CI_TEMPLATE_REGISTRY_HOST/gitlab-org/ci-cd/codequality:0.96.0
    - docker run --rm \
        --volume "$SOURCE_CODE":/code \
        --volume /var/run/docker.sock:/var/run/docker.sock \
        $CI_TEMPLATE_REGISTRY_HOST/gitlab-org/ci-cd/codequality:0.96.0 /code
  artifacts:
    reports:
      codequality:
        - gl-code-quality-report.json
    expire_in: 1 week
  rules:
    - if: '$CI_COMMIT_TAG || $CI_COMMIT_BRANCH'
      when: always
```

`allow_failure: true` гарантирует, что проблемы Code Quality не блокируют слияние.

### 6.4 Специальные стадии `.pre` и `.post`

GitLab поддерживает две специальные стадии:
- `.pre` — выполняется **до** всех остальных стадий
- `.post` — выполняется **после** всех остальных стадий

```yaml
stages:
  - ".pre"
  - test
  - build
  - deploy
  - ".post"

include:
  - component: gitlab.com/gitlab-components/code-quality/code-quality@1.0

code_quality:
  stage: ".pre"
```

Размещение Code Quality в `.pre` даёт ранний фидбэк ещё до запуска тестов.

### 6.5 HTML-отчёт вместо JSON

```yaml
code_quality:
  stage: ".pre"
  variables:
    REPORT_FORMAT: html
  artifacts:
    paths:
      - gl-code-quality-report.html
    reports:
      codequality: []
```

HTML-отчёт удобен для визуального просмотра. Группирует находки по категориям:
- **Bug Risk** — потенциальные баги
- **Complexity** — сложные конструкции
- **Duplication** — дублирование кода
- **Style** — стилевые нарушения

## Часть 7. SAST — Static Application Security Testing

### 7.1 Что такое SAST

**SAST (Static Application Security Testing)** анализирует исходный код, Kubernetes YAML и Helm charts на уязвимости **до** деплоя. GitLab поддерживает несколько анализаторов:

| Анализатор | Назначение | Примечания |
|---|---|---|
| NodeJsScan | Поиск уязвимостей Node.js | `pip install njsscan==<version>` |
| Semgrep | Паттерн-ориентированные статические проверки | Встроен в шаблон |
| KubeSec | Анализ Kubernetes manifest YAML | Встроен в шаблон |

### 7.2 Включение SAST через шаблон

```yaml
include:
  - template: Jobs/SAST.gitlab-ci.yml
```

GitLab **автоматически определяет языки** в репозитории и инжектирует соответствующие задания. Например, при наличии `.js` файлов запускается `semgrep-sast`.

### 7.3 Сканирование Kubernetes манифестов

```yaml
variables:
  SCAN_KUBERNETES_MANIFESTS: "true"
```

Эта переменная инжектирует задание `kubesec-sast` для проверки YAML манифестов.

### 7.4 Полная конфигурация с настройкой стадии

```yaml
stages:
  - .pre
  - test

include:
  - template: Jobs/SAST.gitlab-ci.yml

variables:
  SCAN_KUBERNETES_MANIFESTS: "true"

.prepare_nodejs_environment:
  image: node:16
  before_script:
    - npm ci

sast:
  stage: .pre

unit_testing:
  stage: test
  extends: .prepare_nodejs_environment
  script:
    - npm test
  artifacts:
    when: always
    expire_in: 3 days
    paths:
      - test-results.xml
    reports:
      junit: test-results.xml
```

## Часть 8. Secret Detection — обнаружение секретов

### 8.1 Что такое Secret Detection

**Secret Detection** сканирует репозиторий на чувствительную информацию: API-токены, SSH-ключи, пароли — предотвращая случайное раскрытие. Под капотом используется **Gitleaks** — инструмент с более чем 100 встроенными правилами.

### 8.2 Добавление через компонент

```yaml
include:
  - component: gitlab.com/gitlab-components/secret-detection/secret-detection@1.0
    rules:
      - if: $SECRET_DETECTION_DISABLED == "true" || $SECRET_DETECTION_DISABLED == "1"
        when: never
```

### 8.3 Игнорирование конкретных строк

Если нужно сохранить плейсхолдеры или тестовые токены:

```yaml
"A personal token for GitLab will look like glpat-JUST20LETTERSANDNUMB" # gitleaks:allow
```

> ⚠️ Используйте `# gitleaks:allow` осторожно — никогда не подавляйте обнаружение реальных production-секретов.

### 8.4 Сканирование всей истории репозитория

По умолчанию Secret Detection проверяет только diff каждого коммита. Для сканирования всей истории:

```yaml
variables:
  SECRET_DETECTION_HISTORIC_SCAN: "true"
```

### 8.5 Кастомизация правил обнаружения

Gitleaks работает с TOML-конфигурацией. Можно добавить собственные правила:

```toml
title = "Custom Gitleaks Configuration"

[[rules]]
id = "gitlab_personal_access_token"
description = "GitLab Personal Access Token"
regex = '''\bgitlab-[0-9a-zA-Z\-_]{20}\b'''
tags = ["gitlab", "token"]
keywords = ["glpat"]

[[rules]]
id = "Twilio API Key"
description = "Twilio API Key"
regex = '''5[0-9A-F]{32}'''
keywords = ["twilio", "SK"]

[[rules]]
id = "Facebook token"
description = "Facebook token"
regex = '''(?![0-9]{0,25})(Facebook[a-z0-9-_]{0,5}):([a-z0-9]{32})'''
secretGroup = 2
keywords = ["facebook"]
```

### 8.6 Полная конфигурация задания Secret Detection

```yaml
stages:
  - .pre
  - test

secret_detection:
  stage: .pre
  image: "$CI_TEMPLATE_REGISTRY_HOST/security-products/secrets:5"
  services: []
  allow_failure: true
  variables:
    GIT_DEPTH: '50'
    SECRET_DETECTION_EXCLUDED_PATHS: ''
    SECRET_DETECTION_HISTORIC_SCAN: 'true'
  artifacts:
    reports:
      secret_detection: gl-secret-detection-report.json
  rules:
    - if: '$CI_COMMIT_BRANCH'
      when: always
  script:
    - /analyzer run
```

## Часть 9. Container Scanning — сканирование Docker-образов

### 9.1 Что такое Container Scanning

**Container Scanning** — часть Software Composition Analysis (SCA). Он проверяет Docker-образы и их базовые слои на известные уязвимости. GitLab использует **Trivy** по умолчанию (или GRype).

### 9.2 Включение Container Scanning

```yaml
include:
  - template: Security/Container-Scanning.gitlab-ci.yml

container_scanning:
  variables:
    CS_DEFAULT_BRANCH_IMAGE: $CI_REGISTRY_IMAGE/$CI_DEFAULT_BRANCH:$CI_COMMIT_SHA
```

**Сканирование конкретного образа:**

```yaml
include:
  - template: Security/Container-Scanning.gitlab-ci.yml

variables:
  SECURE_LOG_LEVEL: debug

container_scanning:
  variables:
    CS_IMAGE: example.com/user/image:tag
```

**Аутентификация в приватном registry (AWS ECR):**

```yaml
container_scanning:
  before_script:
    - apk add --no-cache python3 py3-pip
    - pip3 install awscli
    - export AWS_ECR_PASSWORD=$(aws ecr get-login-password --region us-east-1)
    - docker login -u AWS -p "$AWS_ECR_PASSWORD" <aws_account_id>.dkr.ecr.us-east-1.amazonaws.com
  variables:
    CS_IMAGE: <aws_account_id>.dkr.ecr.us-east-1.amazonaws.com/<image>:<tag>
    CS_REGISTRY_USER: AWS
    CS_REGISTRY_PASSWORD: "$AWS_ECR_PASSWORD"
```

### 9.3 Переменные конфигурации

| Переменная | По умолчанию | Описание |
|---|---|---|
| `CS_DEFAULT_BRANCH_IMAGE` | `$CI_REGISTRY_IMAGE/$CI_DEFAULT_BRANCH:$CI_COMMIT_SHA` | Тег образа для дефолтной ветки |
| `CS_IMAGE` | — | Кастомный образ для сканирования |
| `CS_REGISTRY_USER` | — | Username для приватного registry |
| `CS_REGISTRY_PASSWORD` | — | Пароль/токен для авторизации |
| `SECURE_LOG_LEVEL` | info | Детальность логов (`debug` или `info`) |

### 9.4 Allowlist уязвимостей

```yaml
# vulnerabilities-allowlist.yaml
generalAllowlist:
  CVE-2019-8696:
  CVE-2014-8166: cups
  CVE-2017-18248:

images:
  registry.gitlab.com/gitlab-org/security-products/dast/webgoat-8.0@sha256:
    CVE-2018-4180:
```

> ⚠️ Allowlisting пометит CVE как одобренные и они не будут блокировать пайплайн. Используйте с осторожностью.

### 9.5 Интеграция в полный пайплайн

```yaml
stages:
  - .pre
  - test
  - containerization
  - dev-deploy
  - stage-deploy

include:
  - component: gitlab.com/gitlab-components/code-quality/code-quality@1.0
  - template: Jobs/SAST.gitlab-ci.yml
  - component: gitlab.com/gitlab-components/secret-detection/secret-detection@1.0
  - template: Security/Container-Scanning.gitlab-ci.yml

variables:
  DOCKER_USERNAME: siddharth67
  IMAGE_VERSION: "$CI_PIPELINE_ID"
  DOCKER_IMAGE: $DOCKER_USERNAME/solar-system:$IMAGE_VERSION
  CS_ANALYZER_IMAGE: "$CI_TEMPLATE_REGISTRY_HOST/security-products/container-scanning:6"
  CS_SCHEMA_MODEL: 15

container_scanning:
  stage: containerization
  needs:
    - docker_push
  image: "$CS_ANALYZER_IMAGE$CS_IMAGE_SUFFIX"
  variables:
    CS_IMAGE: docker.io/$DOCKER_USERNAME/solar-system:$IMAGE_VERSION
    GIT_STRATEGY: none
    allow_failure: true
  artifacts:
    reports:
      container_scanning: gl-container-scanning-report.json
      cyclonedx: "**/gl-sbom-*.cdx.json"
```

### 9.6 CycloneDX SBOM — Software Bill of Materials

Container Scanning генерирует артефакт SBOM в формате CycloneDX:

```json
{
  "bomFormat": "CycloneDX",
  "specVersion": "1.4",
  "component": {
    "type": "container",
    "name": "docker.io/siddharth67/solar-system",
    "version": "1.1.1"
  },
  "tools": [
    { "vendor": "aquasecurity", "name": "trivy", "version": "0.48.3" }
  ]
}
```

Для демонстрации уязвимостей: смена базового образа с `node:18-alpine3.11` на `node:18` (полный Debian) резко увеличит количество найденных CVE — Alpine намеренно минималистичен и содержит меньше уязвимых пакетов.

## Часть 11. Мониторинг пайплайнов с Prometheus и Grafana

### 11.1 Встроенная аналитика GitLab

GitLab предоставляет встроенный дашборд CI/CD Analytics с:
- Общим процентом успешных пайплайнов
- Трендами длительности пайплайнов за последние 30 коммитов

Но это решение не масштабируется для десятков проектов.

### 11.2 Централизованный мониторинг через GitLab CI Pipelines Exporter

**GitLab CI Pipelines Exporter** опрашивает GitLab API, преобразует метрики пайплайнов, заданий и окружений в формат Prometheus и предоставляет три готовых Grafana-дашборда.

**Варианты установки:**

| Платформа | Команда установки |
|---|---|
| macOS (Homebrew) | `brew install mvisonneau/tap/gitlab-ci-pipelines-exporter` |
| Docker | `docker run -it --rm quay.io/mvisonneau/gitlab-ci-pipelines-exporter:latest` |
| Windows (Scoop) | `scoop install gitlab-ci-pipelines-exporter` |

**Быстрый старт с Docker Compose:**


## Итоговые выводы

**1. DRY — основной принцип оптимизации пайплайнов.** `extends` для кросс-файлового наследования, YAML Anchors внутри файла, `!reference` для точечного импорта — каждый инструмент имеет свою нишу.

**2. Скрытые задания (`.`) — строительные блоки масштабируемых пайплайнов.** Они не выполняются сами по себе, но служат шаблонами для десятков заданий.

**3. `include` превращает один пайплайн в платформу.** `local`, `remote`, `project`, `template` — четыре способа подключить готовую логику извне.

**4. Безопасность — не опция, а обязательная часть пайплайна.** Code Quality, SAST, Secret Detection, Container Scanning — каждый инструмент закрывает свой вектор угроз. `allow_failure: true` делает их информативными, не блокирующими.

**5. Secret Detection — последняя линия защиты от утечек.** Регулярное сканирование всей истории (`SECRET_DETECTION_HISTORIC_SCAN: "true"`) выявляет секреты, случайно закоммиченные ранее.

**6. Container Scanning + SBOM — полная картина уязвимостей образа.** Alpine-образы значительно безопаснее полных Debian-образов — это важный аргумент при выборе базового образа.

**7. Slack + ChatOps = DevOps в мессенджере.** Нотификации дают пассивный мониторинг, ChatOps — активное управление. Вместе они сокращают переключение контекста.

**8. Prometheus + Grafana — для организаций с множеством проектов.** Встроенная аналитика GitLab хороша для одного проекта; Exporter даёт глобальный вид через единый дашборд.

**9. Бейджи — первая точка доверия к проекту.** Зелёный "passed" и "90%" покрытия прямо под описанием репозитория говорят о качестве кода без единого клика.

# Self-Managed Runners — Установка, настройка и практика

---

## Часть 1. Self-Managed Runners — концепция и преимущества

### 1.1 Что такое Self-Managed Runners

**Runners** — это виртуальные машины, которые выполняют задания в GitLab CI/CD пайплайне. Они клонируют репозиторий, устанавливают зависимости и выполняют команды сборки, тестирования и деплоя.

GitLab предлагает два варианта:
- **GitLab-hosted (Shared) Runners** — управляемые GitLab, доступны сразу без настройки
- **Self-Managed Runners** — вы разворачиваете и контролируете собственную инфраструктуру

### 1.2 Преимущества Self-Managed Runners

| Преимущество | Описание |
|---|---|
| Полностью настраиваемое окружение | Выбор ОС, инструментов, библиотек — без ограничений hosted-раннеров |
| Контроль безопасности | Изолированная среда для чувствительных данных и compliance-требований |
| Без очередей | Выделенные ресурсы без ожидания в общей очереди |
| Масштабируемость | Горизонтальное масштабирование для параллельных воркфлоу |
| Снижение задержек | Географическое размещение рядом с целевыми окружениями |

Hosted runners GitLab удобны для старта, но self-managed дают полный контроль над производительностью, безопасностью и стоимостью.

---

### 1.3 Три уровня (scope) раннеров

| Scope | Доступность | Сценарий использования | Место регистрации |
|---|---|---|---|
| Shared runner | Все группы и проекты | Общие CI/CD задачи | Admin Area > CI/CD > Runners |
| Group runner | Все проекты в конкретной группе | Командное разделение ресурсов | Group Settings > CI/CD > Runners |
| Project runner | Один проект | Проектно-специфичные пайплайны | Project Settings > CI/CD > Runners |

Выбор уровня зависит от масштаба: project-раннер для специфических нужд одного проекта, group-раннер для команды, shared-раннер для всей GitLab-инстанции.

## Часть 2. Project-Level Runner с Shell Executor

### 2.1 Настройка Project Runner через UI

**Шаг 1:** В проекте перейдите в **Settings > CI/CD > Runners**.

**Шаг 2:** Отключите существующие Shared Runners (для демонстрации).

**Шаг 3:** Нажмите **New project runner** и выберите:
- **OS**: Linux (или Windows/macOS/Docker/Kubernetes)
- **Tags**: например, `NodeJS` — по этим тегам задания будут направляться на конкретный раннер
- **Run untagged jobs**: опционально, для более широкого охвата
- **Protection, pausing** и другие опции

**Что происходит без раннера:**

После отключения Shared Runners и пуша в `main` пайплайн зависает в состоянии **pending** — нет активных раннеров для выполнения. Задание `unit_testing` будет ждать неограниченно долго:

```
This job is stuck because the project doesn't have any runners online assigned to it.
```

Это наглядно демонстрирует, что без зарегистрированного раннера CI/CD не работает.

### 2.2 Установка GitLab Runner на Linux VM

```bash
# 1. Загружаем бинарный файл GitLab Runner
sudo curl -L --output /usr/local/bin/gitlab-runner \
  https://gitlab-runner-downloads.s3.amazonaws.com/latest/binaries/gitlab-runner-linux-amd64

# 2. Делаем исполняемым
sudo chmod +x /usr/local/bin/gitlab-runner

# 3. Создаём выделенного системного пользователя
sudo useradd --comment 'GitLab Runner' --create-home gitlab-runner --shell /bin/bash

# 4. Устанавливаем и запускаем как системный сервис
sudo gitlab-runner install --user=gitlab-runner --working-directory=/home/gitlab-runner
sudo gitlab-runner start
```

**Проверка установки:**

```bash
# Git branch:   16-8-stable
# GO version:   go1.21.5
# Built:        2024-01-18T22:42:25+0000
# OS/Arch:      linux/amd64
```

Используем регистрационный токен из проекта GitLab:

```bash

  --url "https://gitlab.com/" \
  --registration-token "GLRT-xxxxxxxxxxxxxxxxxxxx"
```


### 2.4 Файл конфигурации `config.toml`

Расположен по пути `/etc/gitlab-runner/config.toml`:

```toml
concurrent = 1
check_interval = 3
shutdown_timeout = 0

[session_server]
  session_timeout = 1800

[[runners]]
  name = "nodejs-runner"
  url = "https://gitlab.com/"
  token = "glrt-..."
  executor = "shell"

[runners.cache]
  MaxUploadedArchiveSize = 0
```

| Настройка | Описание |
|---|---|
| `concurrent` | Максимальное количество параллельных заданий (по умолчанию: 1) |
| `check_interval` | Интервал опроса в секундах (по умолчанию: 3) |
| `session_server.session_timeout` | Таймаут web-terminal сессии (в секундах) |
| `[[runners]].executor` | Тип экзекьютора (`shell`, `docker`, `kubernetes` и т.д.) |

Через `config.toml` можно настроить:
- Имя и теги для маршрутизации заданий
- Блок `executor` для конфигурации Docker-образов, томов, сетей
- Ограничения ресурсов: CPU, память, дисковые квоты
- Переменные окружения
- Пути кэша и артефактов

### 3.3 Устранение ошибки Shell-Profile

После первого запуска может возникнуть ошибка:

```
Preparing environment
Running on iac-server...
ERROR: Job failed: prepare environment: exit status 1.
Check https://docs.gitlab.com/runner/shells/index.html#shell-profile-loading
```

**Причина:** скрипты logout или profile очищают консоль. Нужно отредактировать `~/.bash_logout` на VM раннера:

```bash
# ~/.bash_logout: executed when a login shell exits
# if [ "$SHLVL" -eq '1' ]; then
#   [ -x /usr/bin/clear_console ] && /usr/bin/clear_console -q
# fi
```

Закомментируйте строки с `clear_console`, затем закоммитьте изменения и перезапустите пайплайн.

> ⚠️ Модификация shell-профилей на production-раннерах влияет на все задания. Всегда делайте резервные копии файлов перед редактированием.

### 3.4 Установка Node.js на VM раннера

Shell Executor использует окружение VM напрямую — нет изоляции Docker-контейнера. Это означает, что все инструменты должны быть установлены глобально на хосте раннера:

```bash
sudo apt-get update && sudo apt-get install -y ca-certificates curl gnupg

# Добавляем репозиторий NodeSource
curl -fsSL https://deb.nodesource.com/gpgkey/nodesource-repo.gpg.key \
  | sudo gpg --dearmor -o /etc/apt/keyrings/nodesource.gpg

NODE_MAJOR=20
echo "deb [signed-by=/etc/apt/keyrings/nodesource.gpg] \
  https://deb.nodesource.com/node_${NODE_MAJOR}.x nodistro main" \
  | sudo tee /etc/apt/sources.list.d/nodesource.list

sudo apt-get update && sudo apt-get install -y nodejs

# Проверка:
node -v    # v20.x.x
npm -v     # 9.x.x
```

После установки пайплайн успешно выполнит `npm install` и `npm test`.

## Часть 4. Group Runner с Docker Executor

### 4.1 Концепция Group Runner

**Group Runner** доступен **всем проектам** внутри группы GitLab. Это ключевое преимущество перед Project Runner: один раннер обслуживает команду из десяти проектов без необходимости регистрировать его для каждого отдельно.

---

### 4.2 Создание Group Runner через UI

1. Зайдите в целевую группу (например, `demos`)
2. В левой панели: **CI/CD > Runners**
3. В разделе **Group runners** нажмите **New group runner**
4. Заполните детали:

| Поле | Значение |
|---|---|
| Runner description | Docker executor with AWS on Linux |
| Tags | `docker`, `aws`, `linux` |
| Maximum job timeout | 10 минут |

5. Нажмите **Create runner** — GitLab отобразит **registration token**

---

### 4.3 Установка и регистрация Group Runner

Если GitLab Runner уже установлен (например, при наличии Shell Executor) — пропустите установку. Сразу регистрируем:

```bash
gitlab-runner register \
  --url https://gitlab.com \
  --token <YOUR_REGISTRATION_TOKEN>
```

Интерактивные подсказки:

- **Enter a name for the runner**: `aws-docker-runner`
- **Enter an executor**: `docker`
- **Enter the default Docker image**: `ruby:2.7`

Образ по умолчанию можно переопределить для каждого задания через директиву `image:` в `.gitlab-ci.yml`.

---

### 4.4 Файл `config.toml` с двумя раннерами

После регистрации Group Runner в `config.toml` появится вторая запись `[[runners]]`:

```toml
concurrent = 1
check_interval = 0

[[runners]]
  name      = "nodejs-runner"
  url       = "https://gitlab.com"
  id        = 32418121
  token     = "glrt-3iCBsGsPFN6WBGmaps5B"
  executor  = "shell"
  cache_dir = "/home/gitlab-runner/builds"

[[runners]]
  name     = "aws-docker-runner"
  url      = "https://gitlab.com"
  id       = 32418122
  token    = "glrt-hnyKQKHcCoxosWLEssKc"
  executor = "docker"

  [runners.cache]
    MaxUploadedArchiveSize = 0

  [runners.docker]
    tls_verify                   = false
    image                        = "ruby:2.7"
    privileged                   = false
    disable_entrypoint_overwrite = false
    oom_kill_disable             = false
    disable_cache                = false
    volumes                      = ["cache"]
    shm_size                     = 0
    network_mtu                  = 0
```

Раздел `[runners.docker]` — специфичная конфигурация Docker Executor:
- `image` — образ по умолчанию для заданий без явного `image:`
- `privileged` — привилегированный режим (нужен для DinD)
- `volumes` — монтируемые тома
- `tls_verify` — проверка TLS для Docker daemon

## Часть 5. Локальный шаблон для загрузки отчётов в S3/MinIO

### 5.1 Сценарий: зачем загружать отчёты в S3

GitLab хранит артефакты ограниченное время (настраивается через `expire_in`). Для долгосрочного хранения тестовых отчётов и их доступности вне GitLab — используем S3-совместимое хранилище. **MinIO** — это open-source объектное хранилище, совместимое с AWS S3 API, идеальное для on-premises и self-managed сред.

---

### 5.2 Настройка MinIO

MinIO предоставляет браузерный UI. После запуска:

**Вход:** Username `minioadmin`, Password `minioadmin`

**Создание bucket:** `solar-system-reports-bucket`

Изначально bucket пуст — туда будем загружать `test-results.xml` после каждого тестового запуска.

---

### 5.3 Создание локального шаблона

Создаём файл `templates/aws-reports.yml` в корне репозитория (через Web IDE):

```yaml
variables:
  MINIO_URL: https://<MINIO_SERVER_API>:<PORT>
  BUCKET_NAME: solar-system-reports-bucket
  AWS_ACCESS_KEY_ID: minioadmin
  AWS_SECRET_ACCESS_KEY: minioadmin

.reporting_job:
  stage: reporting
  needs:
    - unit_testing
  image:
    name: amazon/aws-cli:latest
    entrypoint:
      - /usr/bin/env
  before_script:
    - ls -ltr
    - mkdir reports-$CI_PIPELINE_ID
    - mv test-results.xml reports-$CI_PIPELINE_ID/
    - ls -ltr reports-$CI_PIPELINE_ID/
  script:
    - aws configure set default.s3.signature_version s3v4
    - aws --endpoint-url=$MINIO_URL s3 ls s3://$BUCKET_NAME
    - aws --endpoint-url=$MINIO_URL s3 cp ./reports-$CI_PIPELINE_ID s3://$BUCKET_NAME/reports-$CI_PIPELINE_ID --recursive
    - aws --endpoint-url=$MINIO_URL s3 ls s3://$BUCKET_NAME
```

**Разбор шаблона:**

`needs: [unit_testing]` — задание запускается только после успешного завершения `unit_testing`, гарантируя наличие `test-results.xml`.

`image: amazon/aws-cli:latest` с переопределённым `entrypoint: [/usr/bin/env]` — официальный образ AWS CLI. Переопределение entrypoint необходимо, так как стандартный entrypoint образа `aws-cli` запускает AWS CLI напрямую, что ломает стандартный механизм GitLab для выполнения shell-команд.

`mkdir reports-$CI_PIPELINE_ID` — создаём директорию с уникальным именем на основе ID пайплайна. Это обеспечивает разделение отчётов между разными запусками.

`aws configure set default.s3.signature_version s3v4` — устанавливает версию подписи для MinIO (обязательно для корректной работы с MinIO).

`aws --endpoint-url=$MINIO_URL` — все AWS CLI команды направляются на MinIO endpoint вместо реального AWS S3.

### 5.4 Интеграция шаблона в главный `gitlab-ci.yml`

```yaml
workflow:
  rules:
    - if: '$CI_COMMIT_BRANCH == "main" || $CI_COMMIT_BRANCH =~ /feature/'
      when: always

stages:
  - test
  - reporting
  - containerization
  - dev-deploy
  - stage-deploy

include:
  - local: 'templates/aws-reports.yml'

variables:
  DOCKER_USERNAME: siddharth67
  IMAGE_VERSION: $CI_PIPELINE_ID
  K8S_IMAGE: $DOCKER_USERNAME/solar-system:$IMAGE_VERSION
  MONGO_URI: 'mongodb://supercluster.d83jj.mongodb.net/superData'
  MONGO_USERNAME: superuser
  MONGO_PASSWORD: $M_DB_PASSWORD
  SCAN_KUBERNETES_MANIFESTS: "true"

.prepare_nodejs_environment:
  image: node:17-alpine3.14
  services:
    - name: siddharth67/mongo-db:non-prod
  rules:
    - when: always

unit_testing:
  stage: test
  extends: .prepare_nodejs_environment
  tags:
    - docker
    - linux
    - aws
  script:
    - npm test
  artifacts:
    when: always
    expire_in: 3 days
    name: Moca-Test-Result
    paths:
      - test-results.xml
    reports:
      junit: test-results.xml

reporting:
  <<: *reporting_job
  tags:
    - docker
    - linux
    - aws
```

**Таблица стадий:**

| Стадия | Назначение |
|---|---|
| test | Запуск юнит-тестов и генерация JUnit отчётов |
| reporting | Загрузка отчётов в S3/MinIO bucket |
| containerization | Сборка и публикация Docker-образов |
| dev-deploy | Деплой в development окружение |
| stage-deploy | Деплой в staging окружение |

### 5.5 Выполнение пайплайна и проверка результатов

**Логи задания `reporting`:**

```bash
$ ls -ltr
$ mkdir reports-$CI_PIPELINE_ID
$ mv test-results.xml reports-$CI_PIPELINE_ID/
$ aws configure set default.s3.signature_version s3v4
$ aws --endpoint-url=$MINIO_URL s3 ls s3://$BUCKET_NAME
$ aws --endpoint-url=$MINIO_URL s3 cp ./reports-$CI_PIPELINE_ID s3://$BUCKET_NAME/reports-$CI_PIPELINE_ID --recursive
upload: reports-1169963093/test-results.xml to s3://solar-system-reports-bucket/reports-1169963093/test-results.xml
$ aws --endpoint-url=$MINIO_URL s3 ls s3://$BUCKET_NAME
PRE reports-1169963093/
```

После успешного выполнения в MinIO Console появится:
```
solar-system-reports-bucket/
  reports-1169963093/
    test-results.xml
```

Каждый запуск пайплайна создаёт новую папку `reports-<pipeline_id>/`, сохраняя историю всех отчётов.

> ⚠️ Если теги задания не совпадают с тегами раннера, задания зависнут в состоянии pending. Убедитесь, что Group Runner онлайн и теги `docker`, `linux`, `aws` настроены.

## Итоговые выводы

**1. Self-Managed Runners — инвестиция в контроль и производительность.** Hosted runners — отличный старт, но при росте требований к безопасности, специфичным инструментам или предсказуемой производительности — self-managed становится необходимостью.

**2. Три уровня scope — три сценария.** Project Runner для изолированных специфических нужд, Group Runner для командного разделения ресурсов, Shared Runner для общеорганизационных задач. Можно комбинировать все три уровня.

**3. Shell Executor — просто, но с ограничениями.** Прямой доступ к файловой системе VM, нет изоляции Docker. Требует ручной установки всех инструментов на хост. Идеален для простых сценариев или когда Docker недоступен.

**4. Docker Executor — золотой стандарт для большинства случаев.** Изоляция через контейнеры, воспроизводимость среды. Устанавливается аналогично Shell, но при регистрации выбирается `docker` executor.

**5. `config.toml` — единая точка конфигурации всех раннеров.** Один файл управляет параллелизмом, кэш-директориями, Docker-настройками и другими параметрами. Каждый `[[runners]]` блок — отдельный зарегистрированный раннер.

**6. Кэш на Shell Executor хранится на VM.** В отличие от Shared Runners, где кэш управляется GitLab, на self-managed Shell Executor кэш физически находится в `cache_dir` на вашей VM. Путь настраивается в `config.toml`.

**7. Локальные шаблоны через `include: local:` — мощный инструмент повторного использования.** Шаблон в `templates/aws-reports.yml` отделяет логику загрузки в S3 от основного пайплайна. Одно изменение в шаблоне — и все проекты, включающие его, получают обновление.

**8. MinIO = S3 для on-premises.** Совместимость с AWS CLI означает: одни и те же команды, одни и те же паттерны, но на собственной инфраструктуре. Идеально для air-gapped сред или когда нежелателен внешний облачный провайдер.

**9. Теги — критически важны для маршрутизации.** Без правильных тегов задание либо попадёт не на тот раннер, либо зависнет pending. Всегда проверяйте совпадение тегов в `.gitlab-ci.yml` и в настройках раннера.
